## Исторический контекст

### Изменения

- Держать этот план только как дальний backlog-черновик без активного implementation scope.
- Не связывать реализацию `deep link` с ближайшими волнами `GTK` и consumer UX, пока не появится отдельный явный запрос на возврат к теме.
- Выбрать и задокументировать собственную URI-схему, например:
  - `subvost-xray-tun://import-link?...`
  - `subvost-xray-tun://import-subscription?...`
- Подготовить desktop-registration для XFCE/Linux:
  - `.desktop` с `MimeType=x-scheme-handler/...`;
  - launcher, который принимает URI и передаёт его текущему backend.
- Разделить два сценария:
  - импорт уже готовой proxy-ссылки;
  - импорт ссылки на subscription endpoint.
- Добавить защиту от некорректных и опасных payload:
  - ограничение допустимых параметров;
  - валидация схемы и длины;
  - отказ от silent execution без явного подтверждения в UI.
- Решить, как deep link будет говорить с UI:
  - через уже запущенный backend;
  - либо через локальный queue/state файл с последующим открытием окна импорта.

## Исторические проверки

- Ручной smoke: открыть deep link из браузера и убедиться, что bundle поднимается и показывает диалог импорта.
- Негативные кейсы: пустой URI, неподдерживаемая команда, слишком длинный payload, попытка передать непарсируемый URL.
- Проверить, что обычный запуск bundle и deep link запуск не ломают друг другу backend и desktop integration.

## Исторические допущения

- Deep link не должен становиться единственным способом импорта; он вторичен по отношению к обычной кнопке `Импортировать`.
- На первом этапе достаточно Linux/XFCE registration; кросс-дистрибутивные packaging edge-cases можно отложить.
- Реализация deep link имеет смысл только после появления нормализованной локальной модели ссылок и подписок.
- Отсутствие deep link не считается продуктовым долгом ближайших этапов и не блокирует переход к `GTK` или user-facing UX.
