## Цель

Подготовить и затем реализовать первый рабочий native-shell для будущего `GTK4` клиента: окно приложения, навигационный каркас, системный трей и минимальное окно настроек без подключения полной runtime-логики.

## Границы

### Входит

- `GtkApplication` и главное окно приложения;
- `GtkStack` / навигационный каркас для страниц `Dashboard / Subscriptions / Log`;
- Linux tray-интеграция с fallback без падения приложения;
- режимы `close-to-tray` и `start minimized to tray`;
- минимальное окно настроек с поддерживаемыми `v1` переключателями;
- отсутствие раннего `pkexec` на старте.

### Не входит

- service-layer для store/runtime/routing;
- реальная backend-проводка действий `Старт`, `Стоп`, `Снять диагностику`;
- рабочие метрики, логи, импорт подписок и routing UI;
- миграция launcher-а проекта на native-клиент.

## Планируемые изменения

### Код

- добавить новый native-shell контур в `gui/` как отдельный набор модулей для окна, трея и настроек;
- собрать главное окно с переключением между тремя placeholder-страницами;
- реализовать tray-меню и show/hide поведение окна;
- добавить shell-настройки для:
  - файлового логирования;
  - закрытия окна в трей;
  - запуска свёрнутым в трей;
  - темы окна;
- предусмотреть stub-action обработчики для пунктов трея, чтобы UI-контур был собран без полной backend-проводки.

### Конфигурация / схема данных / именуемые сущности

- ограничить настройки только `v1`-параметрами shell-уровня;
- не добавлять в этой подзадаче сетевые и системные настройки без готового backend-контракта.

### Документация

- открыть подзадачу в `subtasks/`;
- добавить строку в `knowledge/tasks/registry.md`;
- при реализации синхронизировать статус подзадачи, план и родительскую задачу.

## Риски и зависимости

- tray-поведение на Linux зависит от конкретного desktop environment и наличия совместимого status notifier;
- если в этой подзадаче начать тянуть service-layer, она разрастётся и смешается со следующими этапами;
- плохой fallback трея может сделать окно “пропадающим” в средах без нормальной tray-поддержки;
- настройки нельзя превращать в фиктивный большой preferences-экран раньше времени.

## Критерии завершения

- окно приложения и навигационный shell работают независимо от текущего web UI;
- системный трей есть в поддерживаемом окружении и безопасно деградирует в неподдерживаемом;
- окно настроек существует как минимальный `v1` shell;
- запуск окна не вызывает `pkexec`;
- task-артефакты синхронизированы и проходят localization guard.

## Фактический результат

- реализованы `gui/native_shell_app.py`, `gui/native_shell_shared.py`, `gui/native_shell_tray_helper.py`;

- минимальные shell-настройки сохраняются в общем `gui_settings.json` без конфликта с текущим web GUI;
- installer расширен опциональной установкой `Gtk3/AppIndicator` GI-зависимостей;
