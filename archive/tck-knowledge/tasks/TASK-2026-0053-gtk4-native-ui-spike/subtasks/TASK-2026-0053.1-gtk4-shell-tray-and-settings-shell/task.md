## Цель

Открыть первый реализуемый этап нативного `GTK4`-клиента: собрать каркас desktop-приложения с главным окном, навигацией по трём страницам, системным треем и оболочкой минимальных настроек, не влезая пока в полноценную backend-интеграцию `Dashboard / Subscriptions / Log`.

## Границы

### Входит

- создание native-каркаса `GtkApplication` для будущего клиента;
- главное окно `GTK4` с header/navigation и тремя страницами стека:
  - `Dashboard`
  - `Subscriptions`
  - `Log`
- системный трей для Linux-сценария bundle:
  - показать окно;
  - скрыть окно;
  - `Старт`;
  - `Стоп`;
  - `Снять диагностику`;
  - `Выход`;
- fallback-поведение, если в окружении нет рабочего status notifier / системного трея;
- сценарии `close-to-tray` и `start minimized to tray`;
- отдельное минимальное окно настроек как оболочка под `v1`:
  - файловое логирование;
  - закрытие окна в трей;
  - запуск свёрнутым в трей;
  - тема окна;
- запрет на ранний `pkexec` при открытии окна и при открытии настроек.

### Не входит

- service-layer над store/runtime/parser/routing;
- реальная backend-интеграция действий `Старт`, `Стоп` и `Снять диагностику`;
- рабочее наполнение `Dashboard`, `Subscriptions` и `Log`;
- живая routing-секция и блок `GeoIP/Geosite`;
- переключение основного launcher-а проекта на новый native-клиент.

## Текущий этап

Подзадача завершена. Реальный desktop-smoke в графической `XFCE`-сессии подтвердил два критичных сценария: fallback без трея не прячет окно и не вызывает ранний `pkexec`, а рабочий tray backend через `Ayatana AppIndicator` поддерживает `start minimized to tray`, открытие главного окна, окно настроек и `close-to-tray` без падения приложения. Дополнительно визуальный контракт этого shell-каркаса вынесен в отдельную подзадачу `TASK-2026-0053.1.1`.

## Стратегия проверки

### Покрывается кодом или тестами

- `python3 -m unittest tests.test_native_shell_shared tests.test_native_shell_app tests.test_subvost_store`;
- `python3 -m py_compile gui/native_shell_shared.py gui/native_shell_app.py gui/native_shell_tray_helper.py gui/subvost_store.py`;
- `bash -n *.sh`;
- `bash -n libexec/*.sh`;
- `bash -n lib/*.sh`;
- runtime-smoke `GTK4` окна без трея в реальной графической сессии с `TrayAvailable=false`, видимым главным окном и отсутствием раннего `pkexec`;
- runtime-smoke `GTK4` окна с tray-helper в реальной графической сессии с `TrayAvailable=true`, скрытым стартом по `start_minimized_to_tray`, командами `ShowWindow` / `OpenSettings` и подтверждённым `close-to-tray`.

### Остаётся на ручную проверку

- отдельные ручные проверки для закрытия этой подзадачи больше не требуются;
- единый финальный ручной smoke для launcher-а, runtime и интеграционного поведения собран в родительской `TASK-2026-0053`.

## Критерии готовности

- у проекта появляется отдельный native `GTK4` shell-контур;
- системный трей работает в поддерживаемом окружении и имеет безопасный fallback;
- окно настроек существует как минимальная `v1`-оболочка;
- подзадача не затягивает в себя service-layer и остальную бизнес-логику;
- запись в реестре и task-артефакты синхронизированы.

## Итог

Собран первый рабочий native-shell контур:

- `gui/native_shell_app.py` поднимает отдельный `GTK4` `ApplicationWindow`, stack-навигацию `Dashboard / Subscriptions / Log`, статусную панель и окно минимальных настроек;
- `gui/native_shell_tray_helper.py` запускает отдельный `Gtk3 + Ayatana/AppIndicator` helper для Linux tray без смешивания `GTK3` и `GTK4` в одном процессе;
- `gui/native_shell_shared.py` хранит чистую shell-логику, страницы, настройки, tray-actions и fallback-решения, которые покрыты unit-тестами;
- `gui/subvost_store.py` расширен до совместимого формата `gui_settings.json`, чтобы web GUI не затирал поля native shell;
- `libexec/install-on-new-pc.sh` теперь пытается ставить `Gtk3` и `AppIndicator` GI-зависимости там, где они доступны.

В рамках закрытия подзадачи дополнительно подтверждено, что `GTK4` shell:

- в fallback-режиме без трея остаётся обычным видимым окном и не делает ранний `pkexec`;
- в рабочей `XFCE`-сессии через `Ayatana AppIndicator` стартует скрытым по `start_minimized_to_tray`, публикует D-Bus control interface, открывает окно настроек и после закрытия главного окна остаётся жить в трее;
- не ломает общий `gui_settings.json`, потому что все проверочные сценарии выполнялись с отдельным `XDG_CONFIG_HOME` и показали ожидаемое поведение настроек.

Остаточные риски этой подзадачи ограничены только тем, что runtime-действия пока остаются shell-stub без реальной backend-проводки, а страницы `Subscriptions` и `Log` ещё не наполнены рабочими данными. Эти риски уже лежат в scope следующих этапов родительской `TASK-2026-0053` и не мешают закрытию `TASK-2026-0053.1`.
