# SDD задачи TASK-2026-0053.1.1

## Контракт

Этот документ фиксирует визуальную систему для нативного `GTK4` клиента на основе `Raycast DESIGN.md`, но адаптирует её под настольную утилиту для `xray/TUN`, а не под браузерный productivity-launcher.

## Проблема

После появления первого `GTK4` shell у проекта есть функциональный каркас, но без отдельного visual contract следующие этапы легко расползутся:

- `Dashboard` может уйти в перегруженный экран мониторинга;
- `Subscriptions` может стать таблицей в стиле web-admin;
- `Log` может стать либо бесформенным терминалом, либо вторичным экраном без иерархии;
- настройки и tray-сценарии могут выпасть из общей системы поверхностей и состояний.

Нужна единая тёмная visual system, которая:

- воспринимается как desktop-shell;
- поддерживает плотный operational workflow;
- не конфликтует с `GTK4` look-and-feel;
- оставляет место для статусов runtime, routing и ошибок.

## Цели

- зафиксировать `Raycast` как основной dark reference;
- описать палитру, поверхности, типографику и акцентные состояния;
- развести уровни плотности для `Dashboard`, `Subscriptions`, `Log`, `tray`, `settings`;
- дать достаточно конкретный контракт, чтобы следующие UI-задачи можно было реализовывать без нового выбора стилистики.

## Что не входит

- не копировать `Raycast` пиксель-в-пиксель;
- не превращать приложение в SaaS dashboard;
- не строить ретро-terminal UI как основную тему;
- не добавлять анимационный или glassmorphism-слой, который будет спорить с `GTK4`.

## Визуальные принципы

### 1. Desktop shell, а не web-dashboard

- окно должно ощущаться как нативная утилита для ежедневного использования;
- внешняя рамка, header и основные панели остаются компактными и собранными;
- основная масса интерфейса живёт в 2-3 уровнях поверхностей, а не в россыпи карточек.

### 2. Тёмная плотная среда

- базовый фон глубокий графитовый;
- большие светлые пустоты не допускаются;
- основные панели ближе по тону друг к другу, чем в типовом web UI;
- лог, статус и действия должны читаться в долгой сессии без зрительного шума.

### 3. Акценты только на смысле

- акцент нужен для `selection`, `focus`, `active runtime state`, важных CTA и критичных ошибок;
- акцент не должен красить весь экран;
- у `Dashboard`, `Subscriptions` и `Log` может быть разная насыщенность контента, но не разный brand language.

### 4. Скорость считывания важнее декоративности

- первое, что пользователь считывает: статус runtime, доступность действий, активный узел, ошибки;
- второе: список сущностей и рабочие операции;
- третье: вторичные метки, подсказки и дополнительный контекст.

### Текст

### Акцент и состояния

- `accent/primary`: `#FF6363`

- `success/warning/error/info` не смешивать с brand-accent без причины;
- для ошибок runtime использовать красный state-color, а не brand-accent;
- на `Log`-экране допускается более высокий контраст между `bg/log` и строками текста.

## Типографика

### Семейства

- UI: `Inter`, `SF Pro Text`, системный `sans-serif`
- Log: `MesloLGS NF`, `FiraCode`, системный `monospace`

### Размерная шкала

- заголовок окна / заголовок экрана: `24-28 px`, `700`
- заголовок панели: `16-18 px`, `650-700`
- основной текст: `13-14 px`, `500`
- вторичный текст: `12-13 px`, `450-500`
- метки / бейджи / вспомогательные подписи: `11-12 px`, `500`

### Правила

- заголовки экранов не должны быть чрезмерно крупными;
- веса шрифтов использовать экономно: акцент через контраст тона и поверхности, а не через жирный текст везде;
- лог всегда `monospace` и визуально отделён от обычных operational panels.

## Отступы и форма

### Радиусы

- внешние панели окна: `16-18 px`
- внутренние карточки / групповые секции: `12-14 px`
- кнопки / поля ввода / pill-элементы: `10-12 px`

### Ритм отступов

- внешний отступ страницы: `18-24 px`
- внутренний отступ панели: `16-20 px`
- компактные интервалы между строками: `8-10 px`
- стандартные вертикальные интервалы: `12-16 px`
- интервалы между секциями: `20-24 px`

### Тени

- мягкие, короткие и тёмные;
- тень нужна только для приподнятых поверхностей;
- большие web-like floating shadows запрещены.

## Иерархия поверхностей

### Уровень 0: Оболочка приложения

- общий фон окна;
- слой заголовка и header-области;
- sidebar или навигационный контейнер.

### Уровень 1: Основные панели

- основной operational content на `Dashboard`;
- список подписок и routing-блок на `Subscriptions`;
- логовая рабочая область на `Log`.

### Уровень 2: Приподнятые элементы управления

- контекстные action-groups;
- popover выбора узла;
- компактное окно настроек;
- временные diagnostic banners.

### Правило

- не использовать больше трёх одновременно читаемых уровней поверхностей на одном экране;
- если секция требует четвёртого уровня вложенности, нужно упрощать композицию.

## Контракты экранов

### Dashboard

Назначение:

- дать моментальный ответ “что сейчас с runtime” и “что я могу сделать дальше”.

Композиция:

- верхняя status-strip с runtime state, ownership state и текущим активным узлом;
- слева primary action cluster `Старт / Стоп / Диагностика`;
- справа compact metrics cluster `uptime / rx / tx / tun / DNS`;
- ниже secondary panel с transport/security и пояснениями по блокировкам.

Визуальное правило:

- не превращать экран в дашборд из 8-12 карточек;
- крупная статусная зона важнее набора маленьких KPI.

### Экран подписок

Назначение:

- рабочий экран управления подписками, узлами и routing-профилями.

Композиция:

- верхняя action-row для импорта и refresh;
- основной список подписок и связанных узлов;
- routing-блок отдельной panel-колонкой или нижней секцией, но внутри той же visual system;
- `GeoIP/Geosite` и routing status не прятать в настройки.

Визуальное правило:

- таблица не должна выглядеть как generic admin grid;
- selected subscription и active node должны считываться быстрее, чем вторичные metadata.

### Экран лога

Назначение:

- диагностическая плоскость, а не просто dump text area.

Композиция:

- header с фильтрами, level chips и export/copy actions;
- основная log surface на отдельном более тёмном фоне;
- короткий summary о последней ошибке или состоянии runtime над логом.

Визуальное правило:

- `Log` может быть визуально ближе к `Warp`, но внутри той же палитры и surface hierarchy;
- log text не должен конкурировать по яркости с CTA и critical banners.

### Системный трей

Назначение:

- быстрые операционные действия без открытия полного окна.

Контракт:

- label и menu items короткие и утилитарные;
- иконка и menu styling не должны пытаться быть “брендовыми” сильнее самой shell-системы;
- tray воспринимается как extension основного окна, а не отдельный мини-продукт.

### Окно настроек

Назначение:

- минимальная оболочка shell-уровня.

Контракт:

- узкое компактное окно;
- секции без визуального мусора и псевдо-маркетинговых пояснений;
- только поддерживаемые настройки: `file logging`, `close to tray`, `start minimized`, `theme`.

Визуальное правило:

- settings не должны выглядеть богаче основных operational screens;
- это утилитарная elevated surface, а не preferences center.

## Компоненты

### Buttons

- primary CTA: акцентный красноватый тон, компактная высота, без giant pill look;
- secondary CTA: тёмная surface с заметной рамкой;
- dangerous action: state/error palette, не brand-accent.

### Inputs

- тёмная плотная заливка;
- focus через accent border и мягкий inner/outer glow;
- placeholder muted, но читаемый.

### Списки и строки

- `hover` и `selected` состояния через изменение поверхности и узкий акцентный маркер;
- активная сущность выделяется не только цветом, но и структурой строки.

### Banners

- только для runtime-blocking и diagnostic conditions;
- короткий заголовок + одна строка действия;
- не занимать много вертикального места.

## Состояния взаимодействия

- `default`: спокойная поверхность панели;
- `hover`: небольшое осветление поверхности;
- `selected`: акцентный маркер и приподнятый фон;
- `focused`: акцентный контур;
- `disabled`: пониженная контрастность, но без полного вымывания;
- `error`: отдельный цвет ошибки и ясный текст причины.

## Доступность и читаемость

- не опускать контраст secondary text ниже комфортного чтения на тёмном фоне;
- focus states обязательны для keyboard navigation;
- critical runtime state должен считываться и по цвету, и по тексту;
- лог и diagnostics должны оставаться читаемыми в долгой сессии.

## Что запрещено

- фиолетовая SaaS-палитра в духе `Linear`;
- светлые карточки на тёмном фоне;
- “hero section” композиция внутри operational app;
- избыточные стеклянные blur-слои;
- чрезмерно игривые градиенты;
- разная visual language у `Dashboard`, `Subscriptions` и `Log`.

## Критерии приёмки

- следующий UI-этап может открыть этот `sdd.md` и получить конкретные правила без повторного выбора референса;
- ручной smoke остаётся единым списком в родительской `TASK-2026-0053`;
- новые `GTK4` экраны могут быть реализованы в одной dark system без конфликтов между shell, tray, settings и diagnostics.
