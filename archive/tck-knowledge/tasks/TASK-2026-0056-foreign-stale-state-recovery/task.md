## Цель

Заменить принадлежность подключения по абсолютному пути на стабильный `install-id` конкретной установки, чтобы перенос production-каталога или деплой на другой ПК не ломал определение владельца из-за старого пути.

## Границы

### Входит

- стабильный install identity в `.subvost/install-id`;
- запись `BUNDLE_INSTALL_ID` в файл состояния вместо `BUNDLE_PROJECT_ROOT` как идентификатора владельца;
- сохранение `BUNDLE_PROJECT_ROOT_HINT` только как диагностической подсказки и резервной логики для старого формата;
- защита живого подключения другой установки;
- явный ручной перехват живого подключения другой установки через кнопку `Перехватить`;
- поддержка переноса каталога той же установки без потери ownership;
- защита production install-id от перезаписи при `install-or-update-bundle-dir.sh`;
- обновление контракта GUI-службы, совместимости запускателя и регрессионных тестов.

### Не входит

- отказ от абсолютных путей как рабочих указателей для wrapper-скриптов, `SUBVOST_PROJECT_ROOT`, `Exec` / `TryExec`, генерируемого конфига и диагностических путей;
- запускатель меню без зависимости от абсолютного пути;
- перенос state-файла из домашнего каталога пользователя;
- автоматический перехват живого подключения другой установки при обычных `Стоп`, shutdown или старте.

## Контекст

- источник постановки: пользователь уточнил, что ожидает честный отказ от абсолютного пути как идентификатора, а не замену старого абсолютного пути новым;
- наблюдаемый симптом: после перезагрузки или запуска production-каталога из другого пути старый файл состояния считался чужим подключением;
- связанная бизнес-область: жизненный цикл переносимой установки, синхронизация production-каталога и переносимость между каталогами/ПК;
- выбранная модель: каждая установленная копия имеет отдельный стабильный `install-id`; dev-копия не управляет production-подключением;
- основной контекст сессии: `новая задача`.

## Критерии готовности

- новый state не использует абсолютный путь как ownership identity;
- перенос каталога той же установки не превращает её подключение в чужое;
- разные установки имеют разные `install-id` и не управляют подключением друг друга;
- кнопка `Перехватить` является единственным явным override для остановки живого подключения другой установки;
- синхронизация production-каталога сохраняет целевой `install-id` и не копирует `install-id` источника;
- устаревший файл состояния с `BUNDLE_PROJECT_ROOT` остаётся поддержанным резервным форматом;
- task-документация и матрица проверки отражают install-id модель.

## Итог

Вместо идентификатора владельца по пути введён локальный для установки `.subvost/install-id`. Shell-сценарии создают или читают этот идентификатор через `subvost_ensure_install_id`, `run` записывает в файл состояния `BUNDLE_INSTALL_ID`, а текущий путь установки сохраняет только как `BUNDLE_PROJECT_ROOT_HINT`. `stop` и GUI-служба классифицируют принадлежность прежде всего по `install-id`; устаревший `BUNDLE_PROJECT_ROOT` остаётся только резервной логикой для старых файлов состояния.

Кнопка `Перехватить` в native GUI теперь вызывает отдельное действие `takeover_runtime`. Оно доступно только для живого подключения другой установки и запускает stop-скрипт с `SUBVOST_FORCE_TAKEOVER=1`; обычные `Стоп`, shutdown и старт сохраняют защиту от случайной остановки чужого runtime.

`install-or-update-bundle-dir.sh` теперь исключает `.subvost` из копирования исходного каталога и вызывает `subvost_ensure_install_id` уже после переключения layout на целевой каталог, чтобы production `install-id` сохранялся между обновлениями и не заменялся `install-id` рабочего checkout-а.

`open-subvost-gui.sh` сравнивает уже запущенную GUI-службу по `gui_version + install_id`: тот же `install-id` и тот же root переиспользуются, тот же `install-id` при другом root ведёт к перезапуску GUI-службы из текущего каталога, другой `install-id` считается несовместимой GUI-службой и уходит в резервный порт.

Локально пройдены `bash -n`, `py_compile` с `PYTHONPYCACHEPREFIX`, `git diff --check`, локализационные guard-ы и `python3 -m unittest discover tests` на 163 теста. Production-каталог синхронизирован после rework, локальный commit переоформлен через `amend`.
