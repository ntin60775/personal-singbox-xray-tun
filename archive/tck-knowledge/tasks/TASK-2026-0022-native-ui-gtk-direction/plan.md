## Исторический контекст

### Изменения

- Зафиксировать, что текущий web UI рассматривается как временный слой, который должен быть заменён на desktop UI поверх уже существующего runtime-контура.
- Зафиксировать выбор `GTK` по причинам:
  - это более нативный путь для Linux/XFCE;
  - он не тащит стек зависимостей `Tauri/WebView`;
  - он лучше совпадает с долгосрочной моделью локального Linux-клиента, чем desktop-обёртка вокруг web UI.
- Отдельно оценить кандидатов для `GTK`:
  - `PyGObject + GTK4` как путь с максимальным переиспользованием Python-стека;
  - `Rust + GTK4/relm4` как более тяжёлый, но сильный вариант для долгосрочного desktop app.
- Зафиксировать, что `Tauri` сейчас в scope не входит.
- Уточнить условия, при которых тема `Tauri` может быть возвращена:
  - появится отдельная задача на мультиплатформенность;
  - решение будет запускаться уже как отдельный `Rust`-проект;
  - перенос в `Tauri` не должен маскироваться под временную миграцию текущего Linux-only клиента.
- Определить целевой split слоёв:
  - data/model/import/subscription logic должна жить отдельно от UI;
  - UI должен быть сменным слоем поверх одного backend-контракта.

## Исторические проверки

- Проверить на MX Linux/XFCE:
  - внешний вид окна;
  - tray/launcher поведение;
  - запуск backend через `pkexec`;
  - открытие диагностики и обработку ошибок.
- Если позже появится отдельный мультиплатформенный spike, сравнивать его нужно уже с готовым `GTK`-направлением, а не с текущим web UI.

## Исторические допущения

- `GTK` считается выбранным основным направлением для desktop UI в этом репозитории.
- Если в будущем появится запрос на мультиплатформенность, `Tauri` допустим только как отдельная ветка развития и сразу в виде `Rust`-проекта.
- До старта `GTK`-реализации сначала всё равно нужно выделить import/subscription/profile слой из текущего backend, чтобы не завязывать новую UI-реализацию на старую web-структуру.
