## Исторические допущения

- `python3` принимается как обязательная системная зависимость проекта, а не как optional runtime.
- Канонизация адресов через стандартный модуль `ipaddress` совместима с адресами TUN, которые bundle ожидает из `sing-box` и `ip -o addr`.

## Исторический итог

Ослаблен DNS preflight: `run-xray-tun-subvost.sh` больше не прерывает старт, если в текущем `/etc/resolv.conf` нет явных `nameserver`, а только предупреждает перед runtime-подменой DNS. Post-start sanity check теперь сравнивает TUN-адреса после канонизации через `python3`/`ipaddress`, поэтому эквивалентные IPv6/IPv4 записи больше не дают ложный rollback. Зависимость `python3` явно оформлена: добавлена ранняя проверка в стартовый скрипт, installer ставит `python3`, README отражает его как обязательную runtime-зависимость bundle.

Статические проверки выполнены: `bash -n *.sh`, `bash -n libexec/*.sh`, `bash -n lib/*.sh`, `python3 -m py_compile gui/gui_server.py`. Остаточный риск: в текущем сеансе не выполнялись ручные `sudo` smoke-проверки старта/остановки стека и негативные сценарии с реальным DNS/TUN-окружением.
