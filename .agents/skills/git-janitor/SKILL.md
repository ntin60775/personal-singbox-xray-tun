# Навык `/git-janitor`

Запуск: просто `/git-janitor`.
Навык не требует входных параметров; если без вопросов нельзя действовать безопасно, ассистент задает минимально необходимые вопросы и продолжает.

## Цель
Навести порядок в git-репозитории:
- при запуске без параметров сначала разобрать синхронизацию текущей ветки с upstream;
- проанализировать текущее состояние (изменения, неотслеживаемые файлы, ветки, upstream);
- разнести изменения по отдельным topic-веткам (по смысловым группам);
- сделать атомарные коммиты;
- слить изменения в integration branch: по умолчанию в `main`, а при подтверждённом flow с `develop` сначала в `develop`, затем в `main`;
- запушить итоговую integration branch, если в репозитории есть валидный upstream и этот `push` вошёл в подтверждённый план janitor-run;
- удалить ставшие ненужными merged-ветки: topic-ветки текущего запуска, а при явном включении в план и исходную рабочую ветку / связанные remote-ветки;
- добиться чистого дерева (`git status` пуст).

## Обязательно

Жёсткие правила безопасности:
- НЕ менять git config.
- НЕ использовать разрушительные команды без явного подтверждения: `reset --hard`, `clean -fd`, `checkout --`, `restore` для потери данных, force-push.
- НЕ коммитить потенциальные секреты: `.env`, `*.key`, `credentials*`, токены в текстах; при подозрении — остановиться и спросить.
- НЕ отключать хуки (флаг --no-verify) без явного запроса.
- При запуске без параметров сначала проверить расхождение текущей ветки с upstream: сперва локальные непушенные коммиты, потом входящие удалённые коммиты.
- Перед любыми командами, которые меняют историю или remote-состояние (`checkout -b`, `commit`, `merge`, `push`, `branch -d`, `push --delete`), показать точный план действий и запросить одно явное подтверждение пользователя на весь janitor-run.
- Если после подтверждения меняются exact-команды, integration branch, remote target или список веток на cleanup, остановиться и запросить новое подтверждение.
- Если план подтверждён и не меняется, отдельный второй вопрос на финальный `push` integration branch и cleanup merged-веток не нужен: это уже часть согласованного janitor-run.

## Базовый сценарий (без аргументов)

### Фаза 0: предварительная синхронизация
Сначала определить состояние текущей ветки относительно upstream:
- `git branch --show-current`
- `git rev-parse --abbrev-ref @{upstream}` (или `NO_UPSTREAM`)
- если upstream есть, обновить remote-tracking refs через `git fetch --prune` нужного remote; если команда требует отдельного согласования/не проходит из-за сети, прямо сказать пользователю, что проверка удалённых изменений основана только на локально известном состоянии upstream
- `git rev-list --left-right --count @{upstream}...HEAD`

Правила интерпретации:
- если `ahead > 0` на не-integration branch, считать эти коммиты входом в janitor-flow; по умолчанию не предлагать отдельный `push` исходной ветки до merge в integration branch.
- если `behind > 0`, показать риск работы на устаревшей базе и спросить, нужно ли сначала подтянуть удалённые изменения через безопасный `git pull --ff-only`;
- если `ahead > 0` и `behind > 0`, явно сообщить, что ветка разошлась; до подтверждённого решения не обещать бесконфликтный full-cycle `push` integration branch.
- если upstream отсутствует, коротко сообщить это и перейти к обычному workflow без отдельного шага `push` для этой ветки; финальный janitor-run тогда фиксируется как local-only.
- если `git pull --ff-only` невозможен, остановиться, показать причину и спросить, как интегрировать расхождение.

После завершения или явного пропуска sync preflight перейти к анализу рабочего дерева.

### Фаза 1: сбор фактов
Выполнить и кратко суммировать:
- `git status --porcelain=v1`
- `git diff --stat` и `git diff --name-only`
- `git diff --staged --stat` и `git diff --staged --name-only`
- `git branch --show-current`
- `git rev-parse --abbrev-ref @{upstream}` (или `NO_UPSTREAM`)
- `git log -30 --pretty=format:%s` (для стиля сообщений)
- проверить наличие `main`/`develop`: `git show-ref --verify refs/heads/main` / `.../develop`

### Фаза 2: выбор целевых веток интеграции
По умолчанию целевая ветка слияния = `main`.
Если `main` отсутствует — спросить, какую ветку использовать.

Если репозиторий использует flow с `develop` (ветка существует и активна), задать 1 вопрос:
1) сливаем только в `main`
2) сливаем в `develop`, затем fast-forward/merge в `main`

### Фаза 3: разнести изменения по workstream-группам
Сформировать группы изменений автоматически:
- сначала по корневым директориям (например, `src/`, `docs/`, `.opencode/`, `.sisyphus/`, `infra/`);
- затем по подсистемам (по следующему уровню пути), если группа слишком большая.

Если есть неоднозначность (например, один файл затрагивает разные темы) — спросить, в какую группу отнести.

### Фаза 3.1: подтвердить план выполнения
Перед модификацией истории выдать пользователю:
- ветка интеграции;
- список workstream-групп с файлами;
- стратегию коммитов и merge;
- точный remote target для итогового `push`, либо явную пометку `local-only`, если upstream недоступен;
- список веток, которые будут удалены после успешной интеграции: локальные и, если применимо, remote-ветки.

После этого спросить подтверждение (`да/нет`) и только при `да` переходить к следующей фазе.

### Фаза 4: создать topic-ветки и атомарные коммиты
Для каждой группы:
1) создать ветку `topic/<slug>` от текущего tip integration branch;
2) перенести в нее только файлы этой группы безопасным способом без потери остальных локальных изменений;
3) если перенос группы нельзя выполнить однозначно и безопасно, остановиться и уточнить у пользователя;
4) разбить на 1+ коммитов:
   - <= 4 файлов на коммит (если больше — обосновать);
   - не смешивать независимые изменения;
   - следовать стилю сообщений из истории (русский/английский, префиксы типа `[tools]`, `docs:` и т.п.).

### Фаза 5: влить topic-ветки в integration branch
Вернуться на integration branch (по умолчанию `main`) и выполнить merge:
- `git merge --no-ff topic/<slug>` (с сохранением истории ветки)
- при конфликтах: остановиться, показать файлы конфликтов, решить минимальным diff.

### Фаза 6: push итоговой integration branch
После локальных коммитов и merge перейти на итоговую integration branch и повторно проверить её состояние относительно upstream:
- `git rev-parse --abbrev-ref @{upstream}` (или `NO_UPSTREAM`)
- `git rev-list --left-right --count @{upstream}...HEAD`

Правила:
- если план подтверждён и upstream доступен, `ahead > 0` после merge означает, что janitor должен выполнить финальный `git push` в рамках уже согласованного сценария;
- если `behind > 0` или fast-forward `push` невозможен, остановиться, показать причину и запросить решение пользователя;
- если upstream отсутствует, коротко сообщить, что janitor не может завершить remote-часть и зафиксировать local-only результат;
- после успешного `push` ещё раз проверить, что расхождение с upstream обнулено или ожидаемо объяснено.

### Фаза 7: cleanup merged-веток
После успешной интеграции и, если применимо, после успешного `push`:
- удалить локальные merged-ветки из подтверждённого плана безопасной командой `git branch -d`;
- удалить remote-ветки только если они были явно включены в подтверждённый план и нет риска затронуть чужую незавершённую работу;
- не удалять integration branch и не удалять ветки, которые не подтверждены как merged / больше не нужны.

### Фаза 8: финальная очистка
В конце гарантировать:
- `git status --porcelain=v1` пуст
- нет оставшихся stash-ей, относящихся к текущей операции

Если остаются изменения, которые нельзя коммитить автоматически (секреты/мусор/временные файлы):
- спросить, что делать: исключить (в .gitignore), коммитить, или удалить/откатить (последнее только с явным подтверждением).

## Итоговый отчёт
В финале вывести:
- какие topic-ветки созданы
- какие коммиты созданы (hash + subject)
- какие merge-коммиты попали в integration branch
- был ли выполнен `push`, в какой remote/branch и чем закончилась повторная проверка расхождения
- какие локальные и remote-ветки удалены в cleanup-фазе
- подтверждение, что `git status` чист
