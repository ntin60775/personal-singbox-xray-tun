# Эталонный тон основного документа

Используй этот файл только как ориентир по тону и степени детализации. Не копируй формулировки механически.

## На что ориентироваться
- Писать так, будто объясняешь доработку разработчику, который откроет конфигуратор после тебя.
- Начинать с предпосылки и эффекта, а не со списка файлов.
- Показывать путь выполнения: где стартует сценарий, где лежит основная логика, какие побочные эффекты важны.
- Выделять хрупкие места и ограничения, если они реально есть.

## Чего избегать
- "В рамках задачи была выполнена доработка механизма..."
- "Также были внесены изменения, направленные на оптимизацию..."
- "Система стала работать лучше и удобнее..."
- Буллетов, которые повторяют заголовок и не добавляют фактов.

## Пример
```markdown
# Контроль статуса внутреннего заказа при загрузке из БУС

## Что менялось и зачем
При загрузке заказа из БУС документ мог создаваться в состоянии, которое дальше не проходило штатную обработку внутреннего заказа. Из-за этого пользователю приходилось вручную искать документ и исправлять статус уже после обмена. Доработка нужна, чтобы после загрузки документ сразу попадал в ожидаемый сценарий обработки и не ломал последующие действия по заказу.

## Предпосылки и контекст
Проблема проявлялась не в UI формы заказа, а в серверной логике обмена. Источник данных - загрузка из БУС, поэтому пользователь видел уже следствие: заказ есть, но дальнейшая цепочка ERP отрабатывает не так, как для штатно созданного документа. Важное ограничение: при загрузке нельзя ломать существующие ветки обмена для других видов заказов.

## Как изменился сценарий
После загрузки из БУС заказ сразу получает тот статус, на который рассчитывает последующая обработка внутреннего заказа. Отдельного ручного исправления после обмена больше не требуется. Для остальных сценариев создания документа поведение не менялось.

## Что изменено в конфигураторе
Точка входа осталась прежней: серверная логика загрузки из БУС. Изменение внесено в участок, где после заполнения объекта фиксируются значения, влияющие на дальнейшую обработку заказа. Дополнительно рядом оставлена проверка, чтобы новый статус применялся только для нужного сценария загрузки и не затрагивал соседние ветки обмена.

## Что важно для сопровождения и доработок
Если дальше потребуется расширять статусы внутреннего заказа, сначала нужно проверить, какие значения ожидают последующие обработчики и отчеты. Хрупкое место здесь - не сам обмен, а согласованность статуса с логикой, которая запускается уже после записи документа. При изменении набора статусов нужно смотреть весь путь, а не только модуль загрузки.

## Проверка
Проверено на сценарии загрузки заказа из БУС с последующим открытием документа в ERP. Дополнительно нужно убедиться, что соседние сценарии обмена, где статус назначается иначе, не изменились.

## Ограничения и риски
В выгрузке не видно отдельного автоматического теста на этот сценарий. Если дальше логика статусов будет расширяться, есть риск, что загрузка и последующая обработка снова разойдутся по ожиданиям.

## Откат и совместимость
Откат сводится к возврату изменений в серверной логике загрузки. Миграций данных нет, но документы, уже созданные с новым статусом, нужно оценивать отдельно, если откат делается после запуска обмена в рабочем контуре.
```
