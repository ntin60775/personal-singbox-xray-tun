# Финальная синхронизация (`diff -> preview -> apply`)

Запускать через `/finalize-sync`.

## Обязательно

### 1) Считать git единственным источником правды
Сначала собрать факты:
- `git status --porcelain=v1`
- `git diff --stat`
- `git diff -U3` по каждому измененному tracked-файлу в индексе и рабочем дереве
- полный список untracked-файлов из `git status`

После этого разложить untracked-файлы по группам:
- `included untracked`:
  - новый файл лежит в той же feature-area, что и tracked diff;
  - новый файл явно продолжает уже измененный объект, модуль, форму или план;
  - новый файл упомянут в активном плане или напрямую нужен для завершения текущего diff;
- `excluded untracked`:
  - временные файлы редактора, кэш, build-артефакты, generated output;
  - файлы из несвязанных каталогов или соседних задач;
  - файлы, связь которых с текущим diff не подтверждается.

Если принадлежность untracked-файла спорная, помещай его в `excluded untracked` и явно показывай это в preview.

### 2) Выбирать план автоматически
Если в репозитории есть `.sisyphus/`:
1. Если существует `.sisyphus/boulder.json` и там указан `active_plan`, используй его.
2. Иначе:
   - если в `.sisyphus/plans/` ровно один файл `*.md`, используй его;
   - если файлов несколько, остановись и уточни у пользователя, какой план синхронизировать.

Если `.sisyphus/` отсутствует, этап синхронизации плана пропускай.

### 3) Всегда строить preview и ждать явного подтверждения
До изменения файлов покажи:
- `tracked changes`: какие tracked-файлы считаются частью финальной правки;
- `included untracked`: какие новые файлы будут учтены и почему;
- `excluded untracked`: какие новые файлы проигнорированы и почему;
- что именно будет синхронизировано в плане;
- какие публичные API-контракты требуют doc-комментария.

Применяй изменения только после явного `да`.

### 4) Синхронизировать план с кодом, а не код с планом
Обновляй выбранный план так, чтобы он соответствовал текущей реализации:
- исправляй пути файлов, имена методов, точки врезки и фактический поток выполнения;
- уточняй deliverables и success criteria только в той части, которая уже подтверждена diff;
- фиксируй расхождения между старым планом и реальным кодом прямо в тексте плана;
- не добавляй новых требований сверху и не делай массовое переформатирование.

### 5) Добавлять doc-комментарии только при реальной необходимости
Ищи новые публичные точки только в реально измененных файлах.

Считай точку кандидатом на комментарий, если выполняется хотя бы одно условие:
- контракт или назначение неочевидны по имени и сигнатуре;
- есть побочные эффекты, запись состояния, запуск обмена или иная интеграционная роль;
- важен порядок вызова, ограничения по входным данным, идемпотентность или условия использования;
- точка служит внешней границей между подсистемами, API или сценариями.

Не добавляй комментарии к:
- тривиальным оберткам и делегирующим вызовам;
- геттерам, сеттерам и одношаговым хелперам с очевидным именем;
- публичным точкам, смысл которых полностью читается из локального контекста.

Для 1C/BSL:
- в первую очередь объясняй назначение, контекст вызова и ограничения;
- секции `Параметры:`, `Возвращаемое значение:` и `Пример:` добавляй только когда без них теряется смысл;
- не делай многострочный блок, если хватает одной короткой технической фразы;
- обязательно упоминай побочные эффекты, если процедура что-то записывает, проводит или отправляет наружу.

### 6) Сохранять существующий формат файлов
Если изменен файл с жесткими требованиями по кодировке или переводам строк, сохраняй текущий формат.
Нормализуй переводы строк только если реально найдено смешение, которое ломает tooling.

## Запрещено
- Не делать `git commit`, `git merge` или `git push` без явного запроса пользователя.
- Не менять поведение кода вне текущего финального diff.
- Не применять изменения без preview и явного подтверждения.

## Итог
Коротко сообщай:
- какие tracked-файлы вошли в финальную правку;
- какие untracked-файлы включены и исключены;
- что синхронизировано в плане;
- какие doc-комментарии добавлены и почему именно к этим точкам.
