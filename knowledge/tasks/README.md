# Методика ведения задач

## Базовый принцип

Для проекта всё строится вокруг **задачи**.
Если действие меняет код, конфигурацию, схему данных, интерфейсы, прикладные правила, тесты или пользовательскую документацию, связанную с изменением поведения системы, это считается задачей.

Задача является центральным узлом, вокруг которого хранятся:

- карточка;
- план;
- SDD для сложных задач;
- журнал работы;
- решения;
- артефакты;
- передача контекста (`handoff.md`);
- подзадачи.

## Что не считается задачей

Следующие действия по умолчанию не открывают режим задачи:

- сборка из исходников;
- создание или доработка служебных скриптов для сопровождения задач;
- технические операционные действия без изменения продукта;
- вспомогательные процедуры среды, если они не являются самостоятельным изменением продукта.

Такие действия относятся к `knowledge/operations/`.

## Явная активация режима задачи

Режим задачи должен быть явно активирован, если в формулировке пользователя есть слово `Задача`.

Даже если слова `Задача` нет, режим задачи всё равно обязателен, когда запрос затрагивает:

- код;
- конфигурацию;
- схему данных;
- интерфейсы;
- интеграции;
- тесты;
- документацию, описывающую изменение поведения продукта.

## Контекст по умолчанию

По умолчанию основным контекстом считается **текущая активная задача**.

Это значит:

- формулировки вроде `добавь ещё`, `исправь здесь`, `в этой задаче` относятся к текущей задаче;
- если пользователь явно указывает другой `ID`, приоритет у указанного `ID`;
- если формулировка может относиться и к текущей задаче, и к другой сущности, нужно явно уточнить, что имеется в виду;
- если активной задачи нет, но запрос уже является задачей, нужно открыть новую внутреннюю задачу формата `TASK-YYYY-NNNN`.

## Автоматическое решение по контексту

Система не должна ждать явного указания `создай задачу` или `создай подзадачу`, если из локального контекста уже ясно, как именно нужно оформить работу.

### Когда продолжать текущую задачу

- запрос остаётся внутри цели и границ активной задачи;
- не возникает отдельный подконтур работы со своим `plan.md`;
- текущая задача остаётся естественным источником истины для статуса, проверок и итога.

### Когда создавать подзадачу

- существует активная родительская задача;
- запрос остаётся внутри цели родителя, но выделяется в самостоятельный этап, компонент, интеграцию, исследование или проверку;
- для этого подконтура полезны собственные статус, `plan.md`, проверки или `handoff.md`;
- родительская задача остаётся основным контейнером цели.

### Когда открывать новую задачу

- меняется основная цель или ожидаемый результат;
- текущая задача перестаёт быть естественным источником истины;
- работа должна сопровождаться независимо и не является частью текущей задачи;
- пользователь явно переключил контекст на другую тему или другой `TASK-ID`.

## Что делать при неоднозначности

Если после локального безопасного исследования нельзя однозначно доказать, что нужен именно один из вариантов, агент обязан остановиться и спросить пользователя.

Вопрос должен:

- кратко объяснять, в чём неоднозначность;
- давать явные варианты действий;
- по возможности использовать формат `A / B / C`.

Предпочтительный набор вариантов:

- `A` — продолжить текущую задачу;
- `B` — создать подзадачу;
- `C` — открыть новую задачу.

## Источник истины

Источником истины по задаче является каталог:

```text
knowledge/tasks/<TASK-ID>-<slug>/
```

Пример внутренней задачи:

```text
knowledge/tasks/TASK-2026-0001-primer-zadachi/
```

Все материалы по задаче должны жить внутри этой папки или явно ссылаться на неё.
`registry.md` нужен только для навигации и не подменяет карточку задачи.

## Идентификаторы

### Основные типы ID

- внешний или бизнес-идентификатор, если он есть;
- внутренний идентификатор проекта: `TASK-2026-0001`.

Если внешний `ID` отсутствует, для внутренней задачи используется формат `TASK-YYYY-NNNN`.

### Подзадачи

Для подзадач допустимы оба варианта:

- отдельный явный `ID`, если он дан пользователем или уже существует;
- наследуемый `ID`, если отдельный `ID` не задан.

Рекомендуемый формат наследуемых подзадач:

- `TASK-2026-0001.1`
- `TASK-2026-0001.1.1`
- `TASK-2026-0001.1.1.1`

Если глубина вложенности превышает 3 уровня подзадач, нужно не создавать следующий уровень молча, а явно предупредить пользователя и предложить перепроектировать декомпозицию.

Подзадача не должна появляться только потому, что запрос упомянул отдельный файл или компонент.
Новая задача не должна появляться только потому, что запрос длинный или содержит несколько шагов.

## Нормализация идентификатора

Одна задача должна использовать единый идентификатор во всех связанных носителях:

| Носитель | Формат | Пример |
|----------|--------|--------|
| ID задачи | внешний или внутренний | `BUG-42`, `TASK-2026-0001` |
| Каталог | `<TASK-ID>-<slug>` | `TASK-2026-0001-primer-zadachi` |
| Ветка Git | `task/...` | `task/task-2026-0001-primer-zadachi` |
| Комментарии в коде | основной ID задачи | `TASK-2026-0001` |
| Именуемые технические сущности | по правилам проекта | project-specific |

Если у задачи нет внешнего бизнес-номера и она требует создания новых именуемых технических сущностей:

- нельзя автоматически выводить новые префиксы только из `TASK-ID`;
- сначала нужно явно зафиксировать в `task.md` технический ключ для новых именуемых сущностей;
- тот же ключ использовать во всех связанных технических именованиях внутри проекта.

Для git-контекста по умолчанию действует следующее:

- верхнеуровневая задача получает ветку `task/<task-id-lower>-<slug>`;
- подзадача по умолчанию наследует ветку родителя;
- если подзадаче нужна отдельная ветка, это должно быть осознанным решением, а не молчаливым отклонением.

## Базовая структура

```text
knowledge/tasks/
├── README.md
├── registry.md
├── _templates/
│   ├── task.md
│   ├── plan.md
│   ├── sdd.md
│   ├── worklog.md
│   ├── decisions.md
│   └── handoff.md
└── TASK-2026-0001-primer-zadachi/
    ├── task.md
    ├── plan.md
    ├── sdd.md
    ├── worklog.md
    ├── decisions.md
    ├── handoff.md
    ├── artifacts/
    │   ├── screenshots/
    │   ├── samples/
    │   └── logs/
    └── subtasks/
        └── TASK-2026-0001.1-primer-podzadachi/
            ├── task.md
            ├── plan.md
            ├── sdd.md
            ├── worklog.md
            ├── decisions.md
            ├── handoff.md
            ├── artifacts/
            └── subtasks/
```

Обязательны для каждой задачи и подзадачи:

- папка задачи;
- `task.md`;
- `plan.md`;
- запись в `registry.md`.

Для сложной задачи дополнительно обязателен `sdd.md`.

## Правило плана

Для этой системы действует жёсткое правило:

- одна задача = один файл `plan.md`;
- одна подзадача = один файл `plan.md`;
- каталоги `plans/` не используются;
- отдельный общий roadmap не нужен: если задача слишком большая, она декомпозируется на подзадачи, каждая со своим `plan.md`.

## Git-контур задачи

- Контур задачи считается незавершённым, пока knowledge-контур и git-контур не синхронизированы друг с другом.
- Поле `Ветка` в `task.md` и колонка `Ветка` в `registry.md` должны совпадать с фактической рабочей веткой.
- На чистом рабочем дереве task-ветка создаётся или переиспользуется автоматически, без отдельного подтверждения пользователя, если контекст задачи ясен.
- Для подзадачи по умолчанию используется ветка родителя; отдельная ветка нужна только если это явно помогает изоляции работы.
- Осмысленные commit-ы по завершении этапа входят в нормальный task-flow и не требуют отдельного напоминания пользователя, если diff относится к одной задаче и не захватывает чужую незавершенную работу.
- Если рабочее дерево грязное несвязанными изменениями, текущая ветка уже занята другой задачей или границы diff неоднозначны, автоматическое git-действие нужно остановить и уточнить стратегию.

## Upgrade старой версии на новую

- Обновление старой версии knowledge-системы или skill-а должно проходить отдельным task-scoped переходом.
- Уже созданные task-каталоги и `knowledge/tasks/registry.md` считаются project data и не должны терять смысл или перезаписываться молча.
- Переход версии должен быть отделён отдельным локальным commit-ом; именно этот commit является видимой точкой перехода в git.
- Проверки перехода должны включать отсутствие дубликатов managed-блока и отсутствие неожиданных изменений в уже созданных задачах.
- Разговор про `push` допустим только если реально обнаружен связанный remote; без remote workflow остаётся локальным.

## Роли файлов внутри задачи

### `task.md`

Главная карточка задачи.
Фиксирует паспорт задачи, цель, границы, контекст, затронутые области, статус, стратегию проверки и итог.

### `plan.md`

Единственный рабочий план задачи.
Содержит шаги реализации, риски, проверки и критерии завершения.
Если для задачи есть `sdd.md`, `plan.md` хранит только исполнимую выжимку по этапам и контрольным точкам без дублирования всей спецификации.

### `sdd.md`

Контракт на реализацию для сложной задачи.
Фиксирует проблему, цель, архитектурное решение, изменения по компонентам, этапы реализации, критерии приёмки и стоп-критерии.

### `worklog.md`

Хронологический журнал движения по задаче.
Сюда попадают факты, блокеры, промежуточные результаты и следующий шаг.
Если есть `sdd.md`, здесь нужно отмечать прохождение этапов, проверки и аудиты.

### `decisions.md`

Фиксирует ключевые решения по задаче, чтобы их не приходилось восстанавливать по коммитам и переписке.
Если решение меняет утверждённый SDD, это нужно зафиксировать явно.

### `handoff.md`

Нужен для передачи задачи другому человеку или пользователю на проверку.

### `artifacts/`

Содержит материалы, которые неудобно держать только текстом:

- скриншоты;
- образцы файлов;
- входные и выходные примеры;
- выгрузки логов;
- постановочные материалы.

### `subtasks/`

Содержит дочерние задачи.
Каждая подзадача повторяет ту же структуру, что и родительская задача.

## Модель статусов задачи

Используется фиксированный набор статусов:

| Статус | Когда ставить | Что это значит |
|--------|---------------|----------------|
| `черновик` | задача создана, но ещё оформляется | структура уже есть, но контекст и границы ещё не зафиксированы окончательно |
| `готова к работе` | постановка уже понятна | задачу можно брать в реализацию |
| `в работе` | идёт анализ, проектирование или реализация | по задаче выполняется активная работа |
| `на проверке` | реализация закончена | нужен контроль результата |
| `ждёт пользователя` | следующий шаг не у исполнителя | нужны данные, решение или ручная проверка |
| `заблокирована` | задачу нельзя двигать дальше | есть внешний блокер |
| `завершена` | задача фактически доведена до конца | всё в границах задачи выполнено и зафиксировано |
| `отменена` | задачу больше не нужно делать | работа остановлена окончательно |

Не использовать статус `готова` без уточнения.
До старта использовать `готова к работе`, после выполнения — `завершена`.

## Когда нужен SDD

`sdd.md` обязателен, если выполняется хотя бы одно условие:

- меняется схема данных, metadata, миграция или конфигурационный контракт;
- меняется внешний API, интеграционный контракт или публичное поведение между подсистемами;
- изменение затрагивает несколько компонентов и должно идти по этапам с отдельными проверками;
- ошибка не может быть надёжно отловлена одной локальной проверкой;
- пользователь явно просит спецификацию, этапность или audit-gates.

`sdd.md` обычно не нужен, если изменение локальное, не меняет контракты и проверяется одним коротким сценарием.
Если сомнение сохраняется, нужно не угадывать, а явно уточнить необходимость SDD.

## Правила хранения связанных данных

### 1. Материалы живут внутри задачи

Нельзя строить систему вокруг общих каталогов вида:

```text
knowledge/screenshots/
knowledge/logs/
knowledge/analysis/
```

Если артефакт относится к задаче, он должен лежать внутри её папки.

### 2. Верхний уровень `knowledge/tasks/` минимален

На верхнем уровне допустимы только:

- `README.md`;
- `registry.md`;
- `_templates/`;
- каталоги задач.

### 3. Статус меняется сначала в карточке задачи

Сначала обновляется `task.md`, потом строка в `registry.md`.

### 4. Если артефакт нельзя положить внутрь задачи, ссылка обязательна

В `task.md` должна быть явная ссылка на внешний объект или связанный ресурс.

## Жизненный цикл задачи

### 1. Инициация

- назначить или получить `ID`;
- создать папку задачи в `knowledge/tasks/`;
- заполнить `task.md`;
- создать `plan.md`;
- для сложной задачи создать `sdd.md`;
- создать или переиспользовать task-ветку;
- записать ветку в `task.md`;
- добавить запись в `registry.md`.

### 2. Анализ

- зафиксировать контекст, ограничения и затронутые области;
- собрать входные материалы;
- если задача сложная, утвердить Problem, Goal, этапы и проверки в `sdd.md`;
- при необходимости декомпозировать задачу на подзадачи.

### 3. Реализация

- обновлять статус и текущий этап;
- синхронизировать ветку в `task.md` и `registry.md`, если она изменилась;
- вести `worklog.md`;
- фиксировать решения в `decisions.md`;
- если есть `sdd.md`, вести реализацию по его этапам и фиксировать отклонения;
- складывать материалы в `artifacts/`.
- по завершении смыслового этапа делать commit в пределах задачи.

### 4. Проверка

- зафиксировать, что покрыто кодом и тестами;
- описать, что остаётся на ручную проверку;
- подготовить handoff при необходимости.

### 5. Завершение или пауза

- обновить итоговый статус;
- зафиксировать результат;
- перечислить остаточные риски;
- при передаче добавить `handoff.md`.

## Правила качества

- одна задача — одна папка;
- одна задача — один `plan.md`;
- подзадача повторяет структуру родителя;
- всё знание, влияющее на сопровождение, должно попасть в файлы задачи;
- build и служебные скрипты сами по себе не считаются задачами;
- при глубине подзадач больше 3 уровней нужно останавливать автоматическую декомпозицию и предлагать перепроектирование.

## Минимальный алгоритм применения

1. Определить, является ли запрос задачей.
2. Если в тексте есть слово `Задача`, сразу активировать режим задачи.
3. Если активна текущая задача и пользователь не указал другой контекст, считать основным контекстом её.
4. Создать или открыть каталог задачи в `knowledge/tasks/`.
5. Создать или синхронизировать task-ветку и сразу записать её в `task.md` и `registry.md`.
6. Если задача сложная, создать и поддерживать `sdd.md`.
7. Поддерживать в актуальном состоянии `task.md` и `plan.md`.
8. При необходимости создавать подзадачи в `subtasks/`.
9. По завершении этапов делать осмысленные commit-ы в пределах задачи.
10. По завершении закрывать задачу через обновление карточки, плана и handoff.
