## Цель

Перевести запуск локального GUI из системного браузера во встроенное desktop-окно на `Python + GTK + WebKitGTK`, не переписывая существующий web UI и не ломая текущий `pkexec`-backend.

## Границы

### Входит

- новый Python-launcher для встроенного `GTK/WebKitGTK` окна поверх существующего `http://127.0.0.1:8421`;
- переключение `open-subvost-gui.sh` с `xdg-open` на новый launcher;
- сохранение безопасного fallback на системный браузер, если desktop webview недоступен;
- обновление installer-зависимостей и launcher-обвязки;
- актуализация task-артефактов и проверок.

### Не входит

- миграция проекта на настоящий `Tauri/Rust` стек;
- переписывание `gui/main_gui.html` в нативные `GTK`-виджеты;
- новая бизнес-логика подписок, runtime или диагностики.

## Контекст

## Стратегия проверки

### Покрывается кодом или тестами

- `bash -n *.sh`
- `bash -n libexec/*.sh`
- `bash -n lib/*.sh`
- `python3 -m py_compile gui/gui_server.py gui/subvost_runtime.py gui/subvost_store.py gui/subvost_parser.py gui/embedded_webview.py`
- `python3 -m unittest tests.test_gui_server tests.test_embedded_webview`
- `python3 gui/embedded_webview.py --check`

### Остаётся на ручную проверку

- fallback на браузер при отключённом `GTK/WebKitGTK`, если потребуется отдельная среда без desktop webview.

## Критерии готовности

- launcher больше не открывает GUI через `xdg-open` по умолчанию;
- встроенное окно поднимается как user-процесс и работает поверх текущего backend;
- при отсутствии `GTK/WebKitGTK` пользователь получает предсказуемый fallback или понятную ошибку;
- installer подготавливает зависимости для desktop webview;
- task-артефакты и реестр синхронизированы с фактической реализацией.

## Итог

Реализован новый launcher `gui/embedded_webview.py`, который поднимает встроенное `GTK/WebKitGTK` окно поверх текущего локального backend-а и поддерживает автоподбор доступного runtime между `Gtk4 + WebKitGTK 6.0` и `Gtk3 + WebKitGTK 4.1/4.0`. `libexec/open-subvost-gui.sh` переведён на режим `auto|webview|browser`: по умолчанию он пытается открыть встроенное окно и только затем делает fallback на системный браузер.

Следом отполирован и сам основной web UI под встроенный desktop-формат: `gui/main_gui.html` получил более собранный control-header, усиленные состояния карточек и кнопок, более читаемую панель подписок/узлов и более чистую визуальную иерархию без изменения backend-логики. По отдельной обратной связи пользователя левая колонка подписок дополнительно расширена под фактическую ширину webview-окна, а действия внутри карточки подписки подняты выше, чтобы блок не выглядел пустым по высоте.

После визуального smoke внесён ещё один проход полировки: polling больше не перерисовывает без необходимости списки и лог при каждом обновлении метрик, поэтому заметное моргание интерфейса убрано. Параллельно усилено выделение активного узла: оно стало заметнее за счёт более точного border/background и мягкого акцентного rail, но осталось в палитре общей тёмной темы.

Дополнительный UI-проход довёл desktop-подачу до более нативного вида: список узлов переложен в плиточную сетку, служебные подсказки убраны под кнопку `?`, а метрики скорости и трафика получили понятные русские подписи вместо `RX/TX`. В следующем визуальном проходе metric-chip были дополнительно расширены и переведены в split-layout с явным разделителем и цветовым акцентом на `вход` / `выход`, а кнопка `Ping` переехала в верхний ряд карточки узла рядом со статусными бейджами. Затем верхние context-pill в шапке были переложены в одну вертикальную колонку, а подсказка лог-панели тоже ушла под знак `?`, чтобы topbar и секция лога стали спокойнее по плотности. Installer теперь ставит `python3-gi` и доступные GI-пакеты для `GTK/WebKitGTK`, не завязываясь жёстко на один namespace. Launcher дополнительно научен подбирать стартовый размер от геометрии монитора: окно сохраняет управляемую ширину, но автоматически раскрывается почти на всю доступную высоту экрана уже при открытии.

Статические проверки и unit-тесты зелёные, а probe `python3 gui/embedded_webview.py --check` в текущей сессии подтвердил `Gtk4 + WebKitGTK 6.0`. Пользователь дополнительно провёл живой smoke встроенного окна в своей графической сессии, подтвердил работоспособность launcher-а и принял итоговую полировку интерфейса.

Остаточный риск: fallback-путь на внешний браузер для окружений без `GTK/WebKitGTK` в этой сессии отдельно не проверялся, поэтому его поведение остаётся опираться на кодовые проверки и существующую shell-обвязку.
