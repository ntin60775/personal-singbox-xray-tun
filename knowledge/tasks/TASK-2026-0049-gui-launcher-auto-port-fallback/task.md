## Цель

Исключить конфликт GUI backend-а между production bundle и запуском GUI из репозитория: если базовый порт GUI занят несовместимым процессом, launcher должен выбрать свободный fallback-порт и открыть интерфейс на нём.

## Границы

### Входит

- Проверка занятости GUI-порта при запуске `open-subvost-gui.sh`.
- Подбор свободного fallback-порта для GUI backend-а.
- Раздельные pid/log-файлы для fallback-портов.
- Тестовое покрытие контракта launcher-а.

### Не входит

- Автоматическое переключение runtime-портов `xray`.
- Изоляция пользовательского store и runtime state.
- Изменение DNS, TUN и правил маршрутизации.

## Контекст

- источник постановки: пользователь подтвердил авто-переключение только для GUI;
- связанная бизнес-область: безопасный запуск dev/prod GUI рядом друг с другом;
- ограничения и зависимости: нельзя останавливать production backend, если он занимает базовый порт;
- основной контекст сессии: `текущая задача`.

## Критерии готовности

- Несовместимый backend на базовом GUI-порту не останавливается.
- Launcher выбирает свободный fallback-порт и открывает frontend на новом URL.
- Проверки проходят.

## Итог

Реализован auto-fallback GUI-порта в `libexec/open-subvost-gui.sh`: launcher проверяет базовый `SUBVOST_GUI_PORT`, переиспользует только совместимый backend того же bundle, а при занятом несовместимом порте выбирает свободный порт до `SUBVOST_GUI_PORT_FALLBACK_END`. Для fallback-портов используются отдельные pid/log-файлы с суффиксом порта, поэтому repo launcher не должен перетирать production GUI pid/log.

Остаточный риск: fallback решает только конфликт GUI backend-а. Общий store и runtime state в `~/.config/subvost-xray-tun` остаются общими до отдельной задачи на dev/prod изоляцию конфигурации.
