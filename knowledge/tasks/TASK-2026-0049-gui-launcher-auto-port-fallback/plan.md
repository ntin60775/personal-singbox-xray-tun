## Цель

Сделать запуск GUI устойчивым к занятому базовому порту, не вмешиваясь в production backend и runtime `xray`.

## Границы

### Входит

- Выбор GUI-порта в `open-subvost-gui.sh`.
- Проверка совместимости уже поднятого backend-а по `gui_version` и `project_root`.
- Fallback на следующий свободный порт при занятом несовместимом порте.
- Разделение pid/log-файлов для fallback-портов.

### Не входит

- Автоматический подбор портов `xray`.
- Перенос или разделение `~/.config/subvost-xray-tun`.
- Смена URL внутри backend API.

## Планируемые изменения

### Код

- Добавить в launcher переменные `PREFERRED_PORT`, `PORT_FALLBACK_END` и динамическое обновление `URL`.
- Перед стартом backend-а искать свободный порт в диапазоне от preferred до fallback-end.
- Сохранять reuse только для совместимого backend-а того же bundle.
- Не вызывать `stop_existing_backend` для несовместимого слушателя на базовом порту.

### Конфигурация / схема данных / именуемые сущности

- Поддержать `SUBVOST_GUI_PORT_FALLBACK_END` как верхнюю границу поиска fallback-порта.

### Документация

- Обновить task-контур и `knowledge/tasks/registry.md`.

## Риски и зависимости

- Fallback-порт решает только конфликт GUI backend-а, но не изолирует общий store/runtime.
- Если весь диапазон занят, launcher должен завершиться понятной ошибкой.

## Итог

`open-subvost-gui.sh` теперь выбирает GUI-порт динамически: базовый порт сохраняется для обычного сценария, а занятый несовместимый порт приводит к поиску свободного fallback-порта. Runtime-порты `xray`, DNS и TUN не менялись.
