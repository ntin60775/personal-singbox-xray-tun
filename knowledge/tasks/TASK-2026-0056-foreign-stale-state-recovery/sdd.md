# SDD задачи TASK-2026-0056

## Назначение

Задача устраняет архитектурную ошибку определения владельца по пути: файл состояния не должен считать абсолютный путь каталога владельцем подключения. После перезагрузки, переноса каталога или деплоя на другой ПК такой путь становится нестабильным и превращает собственное состояние подключения в чужой конфликт.

Новая модель: каждая установленная копия имеет собственный стабильный `.subvost/install-id`. Принадлежность подключения определяется по `BUNDLE_INSTALL_ID`, а абсолютный путь сохраняется только как диагностический `BUNDLE_PROJECT_ROOT_HINT` и резервная логика для старых файлов состояния.

## Инварианты

| ID | Инвариант |
|----|-----------|
| `INV-1` | Постоянный идентификатор владельца подключения не зависит от абсолютного пути каталога. |
| `INV-2` | Перенос каталога той же установки с сохранённым `.subvost/install-id` не превращает подключение или его состояние в чужое. |
| `INV-3` | Разные установки имеют разные `install-id` и не управляют живым подключением друг друга. |
| `INV-4` | Новый файл состояния пишет `BUNDLE_INSTALL_ID` и `BUNDLE_PROJECT_ROOT_HINT`, но не пишет `BUNDLE_PROJECT_ROOT` как ключ владельца. |
| `INV-5` | Устаревший файл состояния без `install-id` остаётся поддержанным через прежнюю резервную проверку пути. |
| `INV-6` | Синхронизация production-каталога не копирует исходный `.subvost/install-id` и сохраняет целевой `install-id` между обновлениями. |
| `INV-7` | Контракт GUI-службы и запускатель используют `install_id` как идентификатор, но перезапускают GUI-службу при смене root той же установки. |
| `INV-8` | Кнопка `Перехватить` является явным пользовательским override и не меняет защиту обычных `Стоп`, shutdown и стартовых сценариев. |
| `INV-9` | Новых runtime/package зависимостей и новых направлений import/module-связей нет. |

## Модульные границы

| Модуль | Ответственность | Допустимые связи |
|--------|-----------------|------------------|
| `lib/subvost-common.sh` | Генерация, чтение и валидация локального `install-id` установки | Не зависит от GUI и менеджера пакетов; использует только базовые системные утилиты с резервными вариантами |
| `libexec/run-xray-tun-subvost.sh` | Предстартовая проверка владельца и запись файла состояния | Использует `subvost_ensure_install_id`; абсолютный путь пишет только как подсказку |
| `libexec/stop-xray-tun-subvost.sh` | Остановка или очистка состояния с защитой чужого подключения | Сначала сравнивает `install-id`; живой чужой runtime останавливает только при `SUBVOST_FORCE_TAKEOVER=1` |
| `libexec/open-subvost-gui.sh` | Выбор или перезапуск GUI-службы | Сравнивает `gui_version`, `bundle_identity.install_id` и root текущей установки |
| `libexec/install-or-update-bundle-dir.sh` | Синхронизация production-каталога | Исключает `.subvost` из копирования источника и создаёт целевой `install-id` после переключения layout |
| `gui/subvost_app_service.py` | Сервисный слой принадлежности и статусный payload | Публикует `bundle_identity.install_id`; содержит отдельное действие `takeover_runtime` |
| `gui/native_shell_app.py` | Native GUI и кнопка ручного перехвата | Кнопка `Перехватить` вызывает service-layer, а не shell напрямую |
| `gui/gui_server.py` | Контекст GUI-службы текущей установки | Передаёт `install-id` в `ServiceContext` |
| `tests/*` | Регрессионный контракт | Проверяет shell-маркеры, сервисный слой и контракт запускателя без root-действий |

Недопустимая связь: UI не должен обходить сервисный слой и самостоятельно решать, можно ли управлять подключением другой установки.

## Критический функционал

- стабильный install-id создаётся локально для конкретной установки и валидируется при чтении;
- живое подключение другой установки не останавливается текущей установкой без явного пользовательского перехвата;
- `Перехватить` останавливает живое подключение другой установки только через `takeover_runtime` и force-флаг stop-скрипта;
- перенос каталога той же установки не ломает ownership;
- production sync не смешивает identity рабочего checkout-а и production-каталога;
- старые state-файлы с `BUNDLE_PROJECT_ROOT` не становятся нечитаемыми.

## Основной сценарий

1. Production-каталог получает или уже имеет `.subvost/install-id`.
2. `run` записывает в state `BUNDLE_INSTALL_ID` и diagnostic `BUNDLE_PROJECT_ROOT_HINT`.
3. После перезагрузки или переноса каталога `stop`, `run` и GUI сравнивают state с текущей установкой по install-id.
4. Если `install-id` совпадает, подключение и его состояние считаются текущими даже при изменившемся абсолютном пути.
5. Если `install-id` отличается и подключение живое, текущая установка отказывает в управлении.
6. Если пользователь нажимает `Перехватить`, GUI вызывает отдельное действие, которое передаёт `SUBVOST_FORCE_TAKEOVER=1` в stop-скрипт.
7. Если `install-id` отличается и живой процесс уже не найден, выполняется очистка устаревшего состояния без остановки чужого процесса.

## Негативные сценарии

- Если `.subvost/install-id` скопирован между разными установками, они начнут считаться одной установкой; поэтому синхронизация исключает `.subvost` из копирования источника.
- Если `.subvost/install-id` потерян при переносе каталога, новая копия будет считаться другой установкой.
- Если GUI launcher найдёт backend той же установки, но из старого root, он должен перезапустить backend из текущего root, а не молча переиспользовать старый процесс.
- Если кнопка `Перехватить` нажата при текущем или уже неактивном подключении, сервисный слой должен отказать без запуска stop-скрипта.

## Наблюдаемый симптом

Пользовательский маркер до исправления:

```text
Файл состояния принадлежит другому bundle
Bundle-владелец runtime: /home/prog7/MyWorkspace/20-Personal/PetProjects/Active/personal-singbox-xray-tun
Текущий bundle: /home/prog7/MyWorkspace/40-Tools-and-Apps/Apps/subvost-xray-tun
```

Новая диагностика должна опираться на идентификатор установки владельца и идентификатор текущей установки, а путь показывать только как последний известный hint.
