## Цель

Убрать двусмысленность в основном пользовательском сценарии `добавил подписку -> увидел результат -> выбрал источник runtime -> запустил`, чтобы web UI был понятен без чтения логов и без знания внутренних файлов bundle.

## Изменения

- Добавить в backend/store явное понятие источника runtime:
  - `store` для сгенерированного конфига из выбранного узла;
  - `builtin` для operator-managed `xray-tun-subvost.json`.
- Довести shell-контракт до совместимого поведения:
  - старт, стоп и диагностика должны одинаково понимать выбранный источник runtime;
  - snapshot активного runtime-конфига должен оставаться доступным для статуса и диагностики.
- Улучшить результат добавления подписки:
  - показать, что подписка действительно сохранена;
  - показать, удалось ли первое обновление;
  - вывести количество импортированных узлов и возможные ошибки без чтения JSON/details.
- Пересобрать web UI вокруг основных задач:
  - активный источник runtime;
  - добавление подписки;
  - подтверждение успешного обновления;
  - явное переключение на встроенный config;
  - ясный выбор профиля и узла.
- Добавить/обновить unit-тесты для runtime-режимов и ответов backend.

## Допущения

- Встроенный config означает локальный operator-managed файл `xray-tun-subvost.json`, который может содержать live-значения вне Git.
- Даже при выбранном `builtin` store с профилями и подписками продолжает существовать: пользователь может готовить/обновлять узлы заранее, не делая их текущим runtime.
- Для этой итерации не требуется выносить HTML/CSS/JS из `gui_server.py` в отдельные ассеты, если текущую связность можно сохранить без ухудшения поддержки.

## Итог

- В backend/store добавлен явный `runtime_preference` с режимами `store` и `builtin`, а shell-слой теперь понимает этот выбор и отражает его в runtime-state без автозапуска стека.
- Web UI перестроен вокруг более ясного сценария: появился заметный переключатель источника runtime, локальный feedback по подпискам и более явные тексты о том, что используется сейчас и что пойдёт в следующий старт.
- На реальной ссылке `https://example.com/subscription` подтвержден безопасный fetch и парсинг без конфликта с текущим продовым VPN: upstream отдал `2` валидные строки в `base64`, после dedup сохранился `1` уникальный узел и это теперь явно показывается в ответе backend/UI вместе с числом схлопнутых дублей.
- Подтверждены безопасные проверки без запуска локального `Старт/Стоп`: unit-тесты, shell syntax-check, `py_compile`, HTTP smoke локального GUI backend, добавление подписки и ручное переключение `builtin/store`.
- Остаточный риск: автоматизированный browser-level smoke через Playwright в этой сессии не выполнен, потому что локальный `playwright-cli` через `npx` зависал на bootstrap; фактическая выдача HTML и HTTP API нового UI/backend подтверждена напрямую через локальный backend.
