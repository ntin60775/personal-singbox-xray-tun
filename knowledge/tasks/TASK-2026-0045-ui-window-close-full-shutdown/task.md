## Цель

Сделать закрытие GUI-окна полным завершением приложения: окно должно закрываться только вместе с GUI-backend и VPN runtime, без висящих фоновых процессов и stale PID-файлов.

## Границы

### Входит

- остановка текущего GUI-backend по запросу пользователя в этой сессии;
- новый shutdown-контракт для embedded GUI-окна;
- остановка VPN runtime при закрытии окна, если runtime действительно активен;
- остановка GUI-backend после успешного завершения VPN runtime или сразу, если runtime уже не активен;
- cleanup PID-файла user-backend при штатном завершении;
- unit-тесты на lifecycle shutdown и cleanup backend-состояния;
- синхронизация task-артефактов и реестра задач.

### Не входит

- изменение поведения browser fallback при закрытии вкладки браузера;
- перепроектирование ручных кнопок `Старт` / `Стоп`;
- изменение сетевой логики VPN runtime вне сценария завершения.

## Контекст

- источник постановки: пользователь явно потребовал, чтобы закрытие окна полностью останавливало UI, GUI-backend и VPN backend;
- связанная бизнес-область: desktop launcher и lifecycle локального VPN bundle;
- ограничения и зависимости: остановка VPN runtime из user-backend идёт через существующий `pkexec`-контур и не должна молча закрывать окно, если stop завершился ошибкой;

## Стратегия проверки

### Покрывается кодом или тестами

- `python3 -m unittest tests.test_gui_server tests.test_embedded_webview`
- `python3 -m py_compile gui/gui_server.py gui/embedded_webview.py gui/gui_contract.py`

### Остаётся на ручную проверку

- открыть embedded GUI, запустить runtime и убедиться, что закрытие окна действительно ведёт к остановке runtime, закрытию GUI-backend и отсутствию висящих PID-файлов;
- отдельно проверить негативный сценарий: если stop runtime завершается ошибкой или отменяется в `pkexec`, окно не должно тихо закрываться с живым VPN backend.

## Критерии готовности

- при закрытии embedded GUI-окна приложение не оставляет живой GUI-backend;
- при активном VPN runtime закрытие окна инициирует его штатную остановку;
- если VPN runtime уже не активен, окно закрывается без лишнего stop-вызова;
- при неуспешной остановке runtime окно не закрывается молча;
- PID-файл GUI-backend очищается при штатном завершении;
- unit-тесты и статические проверки проходят;
- task-контур и реестр задач синхронизированы.

## Итог

В `gui/gui_server.py` добавлен endpoint полного завершения приложения: он определяет, активен ли VPN runtime, и при необходимости прогоняет штатный stop-контур перед остановкой самого HTTP backend. Shutdown backend теперь планируется после ответа клиенту, а cleanup PID-файла привязан к штатному завершению процесса через `atexit`, чтобы после нормального exit не оставался висящий user-backend marker.

В `gui/embedded_webview.py` close handler окна больше не делает прямой `app.quit()`. Теперь закрытие окна сначала отправляет backend-у shutdown-запрос, и только после успешного ответа закрывает GTK-приложение. Если полный stop не удался, окно не закрывается молча: пользователь получает ошибку, а VPN runtime и GUI-backend не теряются из виду.

Тестами зафиксированы новый endpoint завершения, cleanup PID-файла и поведение close handler в сценариях успеха и ошибки. Локально пройдены `python3 -m unittest tests.test_gui_server tests.test_embedded_webview`, `python3 -m py_compile gui/gui_server.py gui/embedded_webview.py gui/gui_contract.py`, `bash -n libexec/open-subvost-gui.sh`, `git diff --check` и `markdown-localization-guard` для новой задачи и реестра.

Остаточный риск: в этой сессии не проигран живой сценарий с реальным `pkexec`-диалогом, поэтому нужно отдельно руками убедиться, что при активном VPN runtime закрытие окна действительно останавливает runtime и затем закрывает окно без зависания.
