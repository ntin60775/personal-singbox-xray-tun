## Цель

Вернуть рабочий запуск embedded web GUI после чёрного экрана и разделить закрытие GUI backend-а от остановки VPN runtime.

## Границы

### Входит

- Отдельный endpoint для закрытия только GUI backend-а.
- Переключение embedded WebKit close-handler на GUI-only shutdown.
- Исправление layout-регрессии после добавления блока маршрутизации.
- Тестовое покрытие нового lifecycle-контракта.

### Не входит

- Переписывание интерфейса на native GTK4.
- Изоляция пользовательского store между dev и prod.
- Изменение `xray` runtime, DNS и TUN.

## Контекст

- источник постановки: пользователь сообщил о чёрном экране после запуска ярлыка и согласовал минимальный repair текущего web GUI;
- связанная бизнес-область: стабильный локальный GUI launcher;
- ограничения и зависимости: production `xray` может работать параллельно, поэтому закрытие GUI не должно автоматически останавливать VPN;
- основной контекст сессии: `текущая задача`.

- Закрытие embedded окна гасит GUI backend, но не вызывает stop runtime.
- Старый endpoint полного закрытия остаётся доступен.
- Основной layout не создаёт отдельную большую строку для routing-блока.
- Проверки проходят.

## Итог

Реализован rescue-патч для текущего web GUI. Добавлен endpoint `POST /api/app/shutdown-gui`, который закрывает только GUI backend и не вызывает остановку VPN runtime. Embedded WebKit close-handler переключён на этот endpoint, а старый `POST /api/app/terminate` сохранён для полного сценария с остановкой runtime.

Layout routing-блока перестроен в трёхколоночную рабочую область: подписки, узлы, маршрутизация. Desktop launcher и installer пользовательского ярлыка теперь задают `SUBVOST_GUI_LAUNCH_MODE=browser` по умолчанию для запуска из ярлыка, чтобы обойти нестабильный embedded WebKit; ручной запуск `./open-subvost-gui.sh` сохраняет режим `auto`, если переменная окружения не задана.

Smoke-проверка через headless Chrome показала, что страница рендерится не чёрным экраном. GUI-only shutdown остановил временный backend и не остановил работающий `xray` runtime. Остаточный риск: живой embedded WebKit без browser fallback отдельно не проверялся.
