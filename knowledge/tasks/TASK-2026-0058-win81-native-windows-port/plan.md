## Цель

Перевести Windows 8.1 порт из состояния архивного prototype в управляемую серию implementation slices: сначала подтвердить архитектуру нативного Windows UI, затем довести build/runtime/docs до проверяемой поставки.

## Границы

### Входит

- фиксация task package и подзадач;
- обсуждаемая архитектура нативного Windows UI;
- план доработки build scripts, runtime hardening, документации и smoke;
- repo-first подход к архивному коду.

### Не входит

- немедленный перенос всего архива в основной repo;
- финальный release zip до реализации и smoke;
- браузерный Windows launcher как целевой UX.

## Риски и зависимости

- главный риск: ошибочный UI-стек. Windows 8.1 старый, поэтому современный webview или актуальные Qt/PySide версии могут не подойти;
- Xray/Wintun compatibility нужно подтверждать конкретной версией, а не словом `latest`;
- build на Linux не доказывает Windows `.exe`;
- Windows runtime меняет default route, поэтому ошибка может оставить пользователя без сети;
- начинающий пользователь не должен читать developer implementation notes вместо нормального README.

## Решение для обсуждения

Рекомендуемый путь для Windows UI: отдельный нативный shell на `.NET Framework 4.8 + Windows Forms`, а Python-часть оставить как core/helper с JSON-командами. Это даёт настоящие Windows-контролы на Windows 8.1 и не требует браузера.

Альтернативы:

- `Python + tkinter`: проще по зависимостям, но UI будет заметно слабее и хуже похож на современное Windows-приложение;
- `PySide2 / Qt5`: визуально сильнее, но тяжелее для Windows 8.1 packaging и добавляет крупную зависимость;
- `PowerShell + Windows Forms`: годится для installer/preflight, но не для основного приложения.

Стек утверждён пользователем: `.NET Framework 4.8 + Windows Forms`.
