## Итог

Усилен `libexec/run-xray-tun-subvost.sh`: перед стартом добавлены проверки `/dev/net/tun`, битых или пустых `resolv.conf`, а для штатных `systemd-resolved`/`NetworkManager` сценариев оставлены предупреждения и диагностика без hard-fail до запуска. После запуска `sing-box` теперь выполняется post-start sanity check по ожидаемому TUN-интерфейсу и всем адресам из текущего `SINGBOX_CONFIG`, без привязки к одному фиксированному внешнему IP или обязательному IPv4-route паттерну; сама проверка ждёт готовность интерфейса polling'ом с таймаутом, следит за живостью и `sing-box`, и `xray`, а не одним снимком сразу после старта. При неполном старте добавлен аварийный rollback с остановкой `xray`/`sing-box`; очистка интерфейсов теперь опирается на pre-start snapshot и адресует только TUN-интерфейс, ожидаемый из текущего конфига, после чего автоматически вызывает `capture-xray-tun-state.sh`.

Расширен `libexec/capture-xray-tun-state.sh`: в единый дамп добавлены секции по `/dev/net/tun`, `lsmod | grep tun`, `ls -l` и `readlink -f` для `/etc/resolv.conf`, состоянию `systemd-resolved` и `NetworkManager`, маршрутам по `tun`/`xray` интерфейсам, а также kernel/systemd-журналам по TUN и DNS-менеджерам. `README.md` дополнен коротким troubleshooting-разделом с фиксированным порядком проверки: TUN-устройство, интерфейс, маршруты, DNS, firewall, MTU.

Статические проверки выполнены: `bash -n *.sh`, `bash -n libexec/*.sh`, `bash -n lib/*.sh`, `python3 -m py_compile gui/gui_server.py`. Остаточные риски: в текущем сеансе не выполнялись ручные smoke-проверки со штатным `sudo ./run-xray-tun-subvost.sh`, негативные сценарии с реальным отсутствием `/dev/net/tun` или конфликтным DNS-менеджером, а также проверка GUI-кнопок и полного диагностического дампа на живой системе.

Исторический Markdown сохранён в `artifacts/legacy-plan.md`; исходный путь legacy-контура зафиксирован в `knowledge/tasks/TASK-2026-0002-plans-to-knowledge-migration/artifacts/legacy/legacy-path-map.md`.
