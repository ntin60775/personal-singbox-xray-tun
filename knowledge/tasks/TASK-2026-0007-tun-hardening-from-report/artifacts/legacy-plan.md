# Заимствования из `docs/research/deep-research-report.md` для hardening bundle

- Дата: 2026-03-27
- Статус: done
- Источник: `docs/research/deep-research-report.md`

## Цель

Усилить bundle за счёт полезных практик из исследовательского отчёта, не превращая проект в клон HAPP и не усложняя штатный сценарий запуска.

Ожидаемый результат: старт и диагностика дают более предсказуемое поведение при проблемах с TUN, DNS и маршрутизацией, а правила проекта фиксируют единый способ документирования таких задач.

## Изменения

- Усилить preflight в публичной команде `run-xray-tun-subvost.sh` через доработку `libexec/run-xray-tun-subvost.sh`: проверка `/dev/net/tun`, ранняя диагностика конфликтов DNS-окружения, более явные сообщения об ошибках до старта `xray` и `sing-box`.
- Добавить post-start sanity check до переключения DNS и записи state-файла: после запуска `sing-box` проверять появление ожидаемого TUN-интерфейса из текущего `SINGBOX_CONFIG`, наличие всех адресов из TUN-inbound и минимальные runtime-инварианты, наблюдаемые в системе после старта, а при сбое завершать старт с понятным сообщением, автоматическим указанием пути к диагностике и rollback частично поднятого состояния (`xray`, `sing-box`, временный `tun`).
- Расширить `capture-xray-tun-state.sh`: добавить сведения о `/dev/net/tun`, `lsmod | grep tun`, `readlink -f /etc/resolv.conf`, состоянии `systemd-resolved` и `NetworkManager`, а также kernel/systemd-журналы по TUN.
- Обновить README коротким troubleshooting-разделом с фиксированным порядком проверки: TUN-устройство, интерфейс, маршруты, DNS, firewall, MTU.
- Не добавлять автоматическую настройку `iptables`, `nftables`, `ip_forward`, `setcap` или постоянную перенастройку системных сетевых сервисов. Эти вещи остаются только частью диагностики и документации.
- Оставить рабочие JSON-конфиги operator-managed; не строить реализацию вокруг их версионируемых миграций, поскольку они исключены из Git.

## Проверки

- `bash -n *.sh`
- `bash -n libexec/*.sh`
- `bash -n lib/*.sh`
- `python3 -m py_compile gui/gui_server.py`
- Ручной happy-path: запуск через `sudo ./run-xray-tun-subvost.sh`, наличие ожидаемого TUN-интерфейса из текущего `SINGBOX_CONFIG`, ожидаемых минимальных runtime-инвариантов после старта, успешная остановка через `./stop-xray-tun-subvost.sh`, корректное восстановление `/etc/resolv.conf`
- Ручной negative-path: понятная ошибка при отсутствии `/dev/net/tun` или невозможности поднять ожидаемый TUN-интерфейс
- Ручной negative-path: понятная ранняя диагностика при конфликте DNS-менеджера или неконсистентном `resolv.conf`
- Ручной negative-path: понятный аварийный выход при неполном старте `sing-box` или провале post-start sanity check, с указанием пути к диагностике и подтверждением rollback частично поднятого состояния
- Проверка, что `capture-xray-tun-state.sh` формирует единый лог с новыми секциями диагностики по TUN, DNS, `systemd-resolved`, `NetworkManager` и журналам

## Допущения

- Bundle остаётся обычным клиентом, а не gateway/router, поэтому автоматический NAT и `FORWARD` не внедряются.
- Rootless-режим не внедряется; штатный запуск по-прежнему идёт через `sudo` и `pkexec`.
- MTU по умолчанию не меняется автоматически; допускается только документированный fallback для ручной диагностики.
- Конфликты с `systemd-resolved` и `NetworkManager` в этой итерации решаются детектом, предупреждением и диагностикой, а не изменением системной конфигурации.
- Критерии sanity check по маршрутизации должны опираться на стабильные runtime-инварианты, наблюдаемые после старта `sing-box` из системных команд (`ip link`, `ip rule`, `ip route`), а не на жёсткую привязку к конкретной локальной версии operator-managed JSON.
- Если post-start sanity check падает до сохранения state-файла, реализация обязана сама остановить поднятые процессы и очистить частично созданное runtime-состояние, чтобы не оставлять orphaned PID, зависший ожидаемый TUN-интерфейс или несогласованный DNS.

## Итог

Усилен `libexec/run-xray-tun-subvost.sh`: перед стартом добавлены проверки `/dev/net/tun`, битых или пустых `resolv.conf`, а для штатных `systemd-resolved`/`NetworkManager` сценариев оставлены предупреждения и диагностика без hard-fail до запуска. После запуска `sing-box` теперь выполняется post-start sanity check по ожидаемому TUN-интерфейсу и всем адресам из текущего `SINGBOX_CONFIG`, без привязки к одному фиксированному внешнему IP или обязательному IPv4-route паттерну; сама проверка ждёт готовность интерфейса polling'ом с таймаутом, следит за живостью и `sing-box`, и `xray`, а не одним снимком сразу после старта. При неполном старте добавлен аварийный rollback с остановкой `xray`/`sing-box`; очистка интерфейсов теперь опирается на pre-start snapshot и адресует только TUN-интерфейс, ожидаемый из текущего конфига, после чего автоматически вызывает `capture-xray-tun-state.sh`.

Расширен `libexec/capture-xray-tun-state.sh`: в единый дамп добавлены секции по `/dev/net/tun`, `lsmod | grep tun`, `ls -l` и `readlink -f` для `/etc/resolv.conf`, состоянию `systemd-resolved` и `NetworkManager`, маршрутам по `tun`/`xray` интерфейсам, а также kernel/systemd-журналам по TUN и DNS-менеджерам. `README.md` дополнен коротким troubleshooting-разделом с фиксированным порядком проверки: TUN-устройство, интерфейс, маршруты, DNS, firewall, MTU.

Статические проверки выполнены: `bash -n *.sh`, `bash -n libexec/*.sh`, `bash -n lib/*.sh`, `python3 -m py_compile gui/gui_server.py`. Остаточные риски: в текущем сеансе не выполнялись ручные smoke-проверки со штатным `sudo ./run-xray-tun-subvost.sh`, негативные сценарии с реальным отсутствием `/dev/net/tun` или конфликтным DNS-менеджером, а также проверка GUI-кнопок и полного диагностического дампа на живой системе.
