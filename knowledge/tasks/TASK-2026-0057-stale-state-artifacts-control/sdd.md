# SDD задачи TASK-2026-0057

## Назначение

Задача стабилизирует жизненный цикл служебных файлов, которые могут остаться после аварийного завершения, неудачного перезапуска или reboot-сценария. Такие файлы не должны бесконтрольно копиться и не должны превращать приложение в тупик, где GUI видит старый state как активный чужой runtime.

Ключевое различие:

- `stale state` — файл состояния есть, но описанный runtime не жив;
- `live state` — state соответствует живому процессу или поднятому TUN-интерфейсу;
- `orphan backup` — DNS backup остался без state и без живого runtime;
- `managed dump` — диагностический файл, созданный приложением по известному шаблону имени.

## Инварианты

| ID | Инвариант |
|----|-----------|
| `INV-1` | Пассивная диагностика не удаляет файлы и остаётся не-мутирующей операцией. |
| `INV-2` | Stale state без живого runtime считается stale artifact, а не активным чужим runtime. |
| `INV-3` | Живой runtime не очищается автоматически; требуется обычная остановка или явный перехват. |
| `INV-4` | DNS backup удаляется только когда state отсутствует и runtime не жив. |
| `INV-5` | Retention cleanup удаляет только обычные файлы известных managed-шаблонов в `logs/`. |
| `INV-6` | Retention window нормализуется и по умолчанию равен `7` дням. |
| `INV-7` | Диагностика показывает state, DNS backup, diagnostic dumps, cleanup availability и причину ручного контроля. |
| `INV-8` | Блок `Диагностика → Файлы и дампы` содержит явное действие `Очистить служебные файлы`. |
| `INV-9` | Диагностические кнопки имеют видимые hover, active, focus и disabled-состояния, а cleanup показывает локальный статус процесса и итог. |
| `INV-10` | Cleanup текущего живого state возвращает понятное no-op сообщение `Очистка не требуется`, а не общий статус частичного выполнения. |
| `INV-11` | Новых runtime/package зависимостей и новых направлений import/module-связей нет. |

## Модульные границы

| Модуль | Ответственность | Допустимые связи |
|--------|-----------------|------------------|
| `gui/subvost_app_service.py` | Аудит и контролируемая очистка runtime artifacts | Может вызывать stop-скрипт только через существующий `run_shell_action`; не удаляет неизвестные файлы |
| `gui/subvost_store.py` | Нормализация и сохранение GUI-настроек | Хранит `artifact_retention_days` без знания runtime-состояния |
| `gui/native_shell_shared.py` | Shared settings и подписи shell actions | Не зависит от GTK и service internals |
| `gui/native_shell_app.py` | Вывод статуса и кнопка cleanup в блоке `Файлы и дампы` раздела `Диагностика` | Вызывает service-layer action, не обходит его |
| `gui/gui_server.py` | HTTP endpoint cleanup | Делегирует в `SubvostAppService.cleanup_runtime_artifacts` |
| `tests/*` | Regression-контракт stale artifacts | Проверяет аудит и cleanup в временных каталогах без root-действий |

Недопустимая связь: UI или HTTP handler не должны самостоятельно удалять state, DNS backup или dumps в обход service-layer.

## Критический функционал

- state классифицируется по фактической живости процесса и TUN-интерфейса;
- cleanup action не трогает живой runtime;
- stale state очищается через существующий stop-сценарий, чтобы использовать один поддерживаемый путь восстановления DNS и интерфейса;
- orphan DNS backup удаляется только после повторной проверки, что state и runtime отсутствуют;
- retention cleanup работает только по двум шаблонам managed-файлов и пропускает symlink или не обычный файл;
- диагностика даёт пользователю наблюдаемый статус вместо скрытого накопления.
- кнопка cleanup визуально реагирует на действия пользователя и показывает локальный feedback выполнения.
- живой state текущего подключения сохраняется и объясняется пользователю как отсутствие требуемой очистки.

## Основной сценарий

1. Пользователь открывает GUI после аварийного завершения или перезагрузки.
2. `collect_status` строит аудит: state есть, процесс и TUN не живы, статус `stale`.
3. Экран диагностики показывает stale state и доступность очистки.
4. Пользователь нажимает `Очистить служебные файлы` или выполняет обычное runtime-действие.
5. UI сразу показывает `Очистить служебные файлы: выполняется.` рядом с кнопками и отключает диагностические действия.
6. Service-layer запускает stop-сценарий для stale state, повторно проверяет runtime и удаляет orphan DNS backup при безопасном условии.
7. Retention cleanup удаляет старые managed dumps, не трогая свежие и неизвестные файлы.
8. Следующий статус показывает отсутствие stale state, количество удалённых managed dumps в `cleanup_summary` и локальный итог выполнения рядом с кнопками.

## Негативные сценарии

- Если state указывает на живой runtime текущего подключения, cleanup action не удаляет state и возвращает сообщение `Очистка не требуется`.
- Если state указывает на живой runtime другого или неподтверждённого экземпляра, cleanup action не удаляет state и возвращает сообщение о ручном контроле.
- Если DNS backup существует, но state или live runtime ещё есть, backup не удаляется.
- Если файл в `logs/` совпал по имени, но является symlink или не обычным файлом, retention cleanup его пропускает.
- Если `artifact_retention_days` повреждён, значение нормализуется к safe default.
- Если пользователь только открыл диагностику, expired dumps не удаляются скрыто.

## Наблюдаемый симптом

Пользовательский маркер до стабилизации:

```text
stale-state воспринимается как активный чужой runtime или остаётся мусорным файлом после аварийного restart.
```

Новая диагностика должна показывать понятный статус: state отсутствует, state текущего подключения, stale state, либо state живого подключения, требующего ручного контроля.
