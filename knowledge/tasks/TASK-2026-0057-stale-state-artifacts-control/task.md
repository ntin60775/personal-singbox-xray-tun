## Цель

Стабилизировать все случаи, где после аварийного или неудачного перезапуска остаются служебные файлы: stale state, orphan DNS backup и старые диагностические дампы. Пользователь не должен получать скрытое накопление мусорных файлов; каждый такой случай должен либо безопасно очищаться программой, либо явно показываться в диагностике рядом с действием очистки.

## Границы

### Входит

- аудит stale state в status payload;
- явное различение живого runtime, stale state и отсутствующего state;
- управляемая очистка stale state через существующий stop-сценарий, если runtime точно не живой;
- очистка orphan DNS backup, если state и runtime отсутствуют;
- retention для управляемых диагностических файлов `xray-tun-state-*.log` и `native-shell-log-export-*.log`;
- вывод статуса служебных артефактов на экран диагностики;
- действие `Очистить служебные файлы` внутри блока `Файлы и дампы` раздела `Диагностика`;
- визуальные состояния кнопок и локальный статус выполнения для диагностических действий;
- REST endpoint для cleanup-действия web/backend контура;
- unit-тесты на аудит, retention и cleanup-действие.

### Не входит

- автоматическая остановка живого подключения другой установки;
- удаление произвольных файлов из `logs/`;
- очистка неизвестных пользовательских файлов;
- перенос state-файла в новый каталог;
- изменение root-модели или `install-id` модели из `TASK-2026-0056`.

## Контекст

- источник постановки: пользователь уточнил, что stale-state не должен копиться после каждого аварийного или неудачного перезапуска;
- ключевой термин: `stale-state` означает файл состояния, который остался на диске, но описанный в нём runtime уже не жив;
- основной риск: автоматическая очистка не должна маскировать живой runtime или удалять неизвестные пользовательские файлы;
- выбранная модель: пассивная диагностика не удаляет файлы, а показывает аудит; фактическая очистка выполняется явным cleanup-действием и в контролируемых runtime-действиях.

## Критерии готовности

- stale state без живого runtime не считается активным чужим runtime и доступен для безопасной очистки;
- живой runtime не очищается автоматически и требует обычной остановки или явного перехвата;
- orphan DNS backup удаляется только когда state и runtime отсутствуют;
- старые управляемые diagnostic dumps не копятся бесконтрольно;
- пассивный статус не удаляет файлы скрыто;
- диагностика явно показывает состояние служебных файлов и доступное cleanup-действие;
- кнопки визуально реагируют на наведение, фокус, нажатие и disabled-состояние;
- после запуска очистки рядом с кнопками виден текущий процесс и итог действия;
- cleanup текущего живого state объясняется как `очистка не требуется`, а не как частичное выполнение;
- unit-тесты и task-документация отражают модель контроля артефактов.

## Итог

Добавлен управляемый аудит служебных артефактов: `collect_status` теперь показывает состояние state, DNS backup, diagnostic dumps, retention window, доступность cleanup и причину ручного контроля. Пассивная диагностика не удаляет файлы.

Добавлено действие `cleanup_runtime_artifacts`: оно не трогает живой runtime, очищает stale state через существующий stop-сценарий, удаляет orphan DNS backup только после повторной проверки отсутствия state/runtime и применяет retention к managed dumps старше `artifact_retention_days`.

Native GUI получил вывод статуса служебных файлов, кнопку `Очистить служебные файлы` и локальный feedback выполнения в блоке `Диагностика → Файлы и дампы`. Backend получил endpoint `/api/artifacts/cleanup`. Настройка `artifact_retention_days` нормализуется в диапазоне `1..365`, default `7`.
