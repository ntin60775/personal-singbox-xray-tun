## Цель

Добавить в bundle альтернативный runtime-вариант, который поднимает TUN-режим только на `xray-core`, без обязательной зависимости от `sing-box`, и при этом не ломает текущий основной сценарий `Xray + sing-box`.

Ожидаемый результат: в проекте появляется отдельный `xray-only` сценарий запуска/остановки с собственным runtime-конфигом и понятными ограничениями, а диагностика, статус GUI и документация не вводят пользователя в заблуждение при работе этого режима.

## Границы

### Входит

- добавление отдельного `xray-only` runtime-конфига и wrapper-скриптов;
- materialize runtime-конфига на основе активного `Xray`-узла с наложением TUN/DNS/routing-слоя;
- адаптация статуса GUI и диагностического дампа под `xray-only` режим;
- обновление документации и knowledge-артефактов.

### Не входит

- удаление существующего runtime на `Xray + sing-box`;
- полная миграция GUI-кнопок запуска на новый режим по умолчанию;
- гарантия полного parity с `sing-box` по auto-route/auto-redirect и всем сценариям DNS;
- полный ручной smoke с реальным трафиком и `sudo` в рамках этой сессии.

## Контекст

- источник постановки: вопрос пользователя, зачем в проекте нужен `sing-box`, и запрос на версию только на `xray-core`
- связанная бизнес-область: локальный portable bundle для proxy/TUN-режима
- ограничения и зависимости: текущий проект исторически опирается на `sing-box` для TUN, DNS hijack и маршрутизации; новый режим должен сосуществовать с ним без регрессии текущего сценария
- основной контекст сессии: `текущая задача`

### Остаётся на ручную проверку

- проверка реального доступа в сеть и DNS-поведения на целевой машине.

## Критерии готовности

- в корне bundle есть отдельный запускаемый `xray-only` сценарий без обязательного `sing-box`;
- runtime-конфиг собирается из активного `Xray`-конфига и валидируется локально;
- stop/diagnostics/GUI-status не считают `xray-only` режим ошибкой по умолчанию;
- README и реестр задач обновлены, а Markdown прошёл localization guard.

## Итог

Добавлен отдельный CLI-сценарий `xray-core only`: новые wrapper'ы `run-xray-only-tun-subvost.sh` и `stop-xray-only-tun-subvost.sh`, tracked-шаблон `xray-only-tun-subvost.json` и shell-логика, которая materialize runtime-конфиг из активного `Xray`-конфига, поднимает `tun0` через `xray-core`, настраивает policy-routing по `fwmark` и корректно откатывает DNS и маршруты через общий state-файл.

Python-слой обновлён так, чтобы `gui/subvost_runtime.py` умел строить `xray-only` runtime-конфиг из активного узла, а `gui/gui_server.py` распознавал `runtime_impl=xray-only`, корректно считал стек рабочим без `sing-box`, показывал правильный `stack_line`, `tun_interface` и не маркировал такой запуск как деградацию по умолчанию.

Диагностика и документация синхронизированы: `capture-xray-tun-state.sh` учитывает `runtime_impl` и динамическую routing-таблицу, `README.md` и `AGENTS.md` объясняют, зачем основной проект использует `sing-box`, где находится новый `xray-only` режим и какие у него ограничения.

Проверки выполнены так: `bash -n *.sh`, `bash -n libexec/*.sh`, `bash -n lib/*.sh`, `python3 -m py_compile gui/gui_server.py gui/subvost_runtime.py`, `python3 -m unittest tests.test_subvost_runtime tests.test_gui_server`, `python3 -m json.tool xray-only-tun-subvost.json`, а также адресный запуск `markdown-localization-guard` для изменённых Markdown-файлов. Дополнительно фактическая сборка нового runtime-конфига проверена на локальном `Xray 26.2.6`: конфиг читается и доходит до этапа создания TUN-сервера, после чего упирается в отсутствие runtime-окружения для реального старта в этой сессии.

Остаточный риск: в текущей среде не выполнен полноценный ручной smoke с `sudo ./run-xray-only-tun-subvost.sh`, проверкой `tun0`, `ip rule`, реального DNS/трафика и последующего отката. Это нужно отдельно подтвердить на целевой Linux-машине.
