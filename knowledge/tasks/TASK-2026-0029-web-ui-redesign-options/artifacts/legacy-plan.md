# Варианты полного редизайна web UI

- Дата: 2026-04-01
- Статус: done
- Источник: запрос пользователя провести полный редизайн и пересборку web UI, сначала накидать варианты дизайнов и показать их

## Цель

Подготовить несколько сильных визуально и UX-смыслово разных направлений для web UI, чтобы сравнить их на живом локальном экране и затем выбрать основу для полной пересборки пользовательского интерфейса.

## Изменения

- Зафиксировать текущие проблемы существующего web UI:
  - перегруженность экрана карточками;
  - слабая визуальная иерархия;
  - неудобное разделение главных действий и вторичных данных;
  - операторские детали слишком рано попадают в основной сценарий.
- Собрать отдельный экран сравнения дизайн-направлений на базе существующего GUI backend:
  - минимум 3 разных направления;
  - единый набор данных и одинаковые ключевые сценарии;
  - акцент на читаемость статуса, запуск/остановку, выбор узла и работу с подписками.
- Для каждого варианта явно показать:
  - визуальный тезис;
  - ключевые UX-решения;
  - сильные стороны и компромиссы.
- Не ломать текущий production-маршрут GUI до момента, пока пользователь не выберет направление для полной пересборки.

## Проверки

- `python3 -m py_compile gui/gui_server.py`
- `python3 -m unittest tests.test_gui_server`
- Ручной локальный HTTP smoke:
  - открыть текущий GUI;
  - открыть страницу сравнения вариантов;
  - убедиться, что все варианты рендерятся;
  - убедиться, что данные состояния и store читаются без падения страницы.

## Допущения

- На этой итерации требуется именно экран вариантов и живой preview, а не финальная замена основного интерфейса.
- Варианты можно собирать в том же `gui/gui_server.py`, если это ускоряет сравнение и не ломает текущий контракт backend.
- После выбора направления пользователем потребуется отдельная итерация на полную интеграцию выбранной концепции в основной маршрут GUI.

## Итог

- В backend добавлен отдельный маршрут `/design-review`, который не ломает текущий основной маршрут GUI и позволяет сравнивать варианты на живых данных из `/api/store`.
- Собраны три разных дизайн-направления:
  - `Signal Deck` как основной кандидат для ежедневной работы;
  - `Guided Flow` как сценарный и более дружелюбный UX;
  - `Ops Console` как плотный операторский интерфейс.
- Добавлен отдельный HTML-asset `gui/design_review.html` с адаптивной страницей сравнения и локальной JS-нормализацией данных.
- Проверено:
  - `python3 -m py_compile gui/gui_server.py`;
  - `python3 -m unittest tests.test_gui_server`;
  - локальный HTTP smoke через поднятый backend и `curl` на `/design-review` и `/api/store`.
- Остаточный риск: browser-level визуальный smoke в реальном графическом браузере в этой сессии не выполнялся; подтверждены Python-проверки и реальная HTTP-отдача страницы вариантов.
