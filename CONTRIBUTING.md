# Как вносить изменения

Репозиторий открыт для `issue` и `pull request`, если изменения помогают держать bundle переносимым, безопасным и понятным для Linux-сценария с `xray-core` в `TUN`-режиме.

## Перед началом

- Для нетривиальной доработки сначала открой issue и опиши цель, ограничения и ожидаемый результат.
- Один pull request должен решать одну проблему, без смешивания GUI-, installer- и сетевых изменений без явной причины.
- Не присылай реальные subscription URL, UUID, `publicKey`, `shortId`, локальные IP, приватные дампы и другие чувствительные runtime-данные.
- Внутренний контур `knowledge/` ведут maintainers. Внешнему контрибьютору не нужно самостоятельно заводить task-артефакты, если это отдельно не согласовано.

## Локальная проверка перед PR

Минимальный набор:

```bash
bash -n *.sh
bash -n libexec/*.sh
bash -n lib/*.sh
python3 -m py_compile gui/gui_server.py gui/subvost_runtime.py gui/subvost_store.py gui/subvost_parser.py gui/embedded_webview.py
python3 -m unittest tests.test_subvost_parser tests.test_subvost_store tests.test_subvost_runtime tests.test_gui_server tests.test_embedded_webview
python3 -m json.tool xray-tun-subvost.json
```

Если меняется маршрутизация, preflight, DNS или диагностика, добавь в описание PR ручной smoke:

```bash
sudo ./run-xray-tun-subvost.sh
ip -brief link show tun0
ip rule show
./stop-xray-tun-subvost.sh
```

Если меняется GUI, приложи скриншот или короткое видео и перечисли, какие действия были проверены: `Старт`, `Стоп`, `Снять диагностику`, импорт подписки, выбор узла.

## Правила для pull request

- Используй короткие commit-prefixes по текущей истории репозитория: `feat:`, `fix:`, `docs:`, `chore:`.
- Держи shell-скрипты в стиле `bash` + `set -euo pipefail`, отступ 2 пробела и говорящие имена функций.
- Для Python придерживайся PEP 8, 4-пробельного отступа и `snake_case`.
- Описывай не только что изменилось, но и какой пользовательский или операционный сценарий это чинит.
- Если изменение влияет на публичный контракт README, installer-а, GUI или runtime-шаблона, обнови документацию в том же PR.

## Что чаще всего разворачивается обратно

- PR без воспроизводимого сценария или без объяснения, зачем менять сетевое поведение.
- Любые коммиты с секретами, приватными URL или несанифицированными логами.
- Большие несвязанные рефакторы без предварительного обсуждения.
- Изменения, которые автоматически меняют постоянную системную конфигурацию DNS, firewall или network-manager без очень сильного обоснования.

## Вопросы по процессу

Если не уверен, стоит ли тащить идею в код, сначала открой issue с коротким описанием идеи. Это дешевле, чем спорить уже по готовому PR.
