# Правила репозитория

## Структура проекта и модулей

Репозиторий представляет собой переносимый bundle для запуска `xray-core` в `TUN`-режиме без дополнительных proxy-движков. Публичные точки входа в корне: `run-xray-tun-subvost.sh`, `stop-xray-tun-subvost.sh`, `capture-xray-tun-state.sh`, `open-subvost-gui.sh`, `install-on-new-pc.sh`, `install-subvost-gui-menu-entry.sh` и `subvost-xray-tun.desktop`. Основной tracked-конфиг лежит в корне: `xray-tun-subvost.json` хранится в Git только как санитизированный шаблон для генерации runtime-конфига выбранного узла и не запускается напрямую. Диагностика и временные логи попадают в `logs/`. Внутренняя shell-реализация вынесена в `lib/` и `libexec/`, GUI backend — в `gui/`, исследовательские материалы — в `docs/research/`, вспомогательные repo-managed артефакты — в `assets/`.

Для task-centric knowledge используется каталог `knowledge/`:

- `knowledge/README.md` — обзор knowledge-системы и её контуров
- `knowledge/tasks/` — каноническое место для рабочих задач и подзадач
- `knowledge/tasks/registry.md` — навигационный реестр задач
- `knowledge/tasks/_templates/` — шаблоны `task.md`, `plan.md` и сопутствующих артефактов
- `knowledge/operations/` — операции среды и сопровождения, которые не считаются задачами продукта

Исторические планы уже мигрированы в `knowledge/tasks/`. Их происхождение и исходный Markdown сохраняются внутри соответствующих task-каталогов, обычно в `artifacts/legacy-plan.md`. Новые задачи, подзадачи и рабочие планы создаются только в `knowledge/tasks/`.

Артефакты knowledge-системы являются частью инженерной документации репозитория и хранятся в Git. `xray-tun-subvost.json` допустим в Git только в санитизированном виде: live-значения `outbounds[0].settings.vnext[0].users[0].id`, `outbounds[0].streamSettings.realitySettings.publicKey` и `outbounds[0].streamSettings.realitySettings.shortId` должны попадать только в локально сгенерированный runtime-конфиг и не коммитятся.

## Команды запуска, проверки и разработки

Сборки как отдельного этапа нет: bundle запускается напрямую.

- `sudo ./run-xray-tun-subvost.sh` — поднять основной `xray` runtime из активного узла, настроить DNS и сохранить state.
- `./stop-xray-tun-subvost.sh` — корректно остановить runtime и восстановить `resolv.conf`.
- `sudo ./capture-xray-tun-state.sh` — снять полный диагностический дамп в `logs/`.
- `./open-subvost-gui.sh` — запустить GUI launcher без раннего root-запроса; действия `Старт`, `Стоп` и `Диагностика` внутри GUI запрашивают root через `pkexec`.
- `bash ./install-on-new-pc.sh` — поставить системные зависимости на Debian/Ubuntu.
- `bash -n *.sh`, `bash -n libexec/*.sh`, `bash -n lib/*.sh`, `python3 -m py_compile gui/gui_server.py gui/subvost_runtime.py gui/subvost_store.py gui/subvost_parser.py` — минимальная локальная проверка перед коммитом.

## Стиль кода и именование

Для shell-скриптов сохраняйте `bash` + `set -euo pipefail`, отступ 2 пробела и говорящие имена функций (`find_executable`, `ensure_absolute_path`). В Python придерживайтесь PEP 8, 4-пробельного отступа и `snake_case` для функций/переменных. Новые скрипты называйте по действию и контексту, по образцу `capture-xray-tun-state.sh`. Не зашивайте пользовательские пути: используйте переменные окружения и абсолютные пути только там, где это уже часть контракта.

Для каталогов задач используйте формат `<TASK-ID>-short-slug`, например `TASK-2026-0001-task-centric-knowledge-rollout`. Содержимое `task.md`, `plan.md` и связанных task-артефактов ведите на русском, с короткими машинно-значимыми идентификаторами только там, где это действительно нужно.

## Порядок работы

Для любой нетривиальной доработки придерживайтесь одного и того же цикла:

1. Сначала собрать факты: текущее поведение скриптов, DNS/TUN-модель, ограничения окружения, следы в логах.
2. Если запрос является задачей, создать или открыть каталог `knowledge/tasks/<TASK-ID>-<slug>/`, поддерживать в актуальном состоянии `task.md` и `plan.md`, а также сразу добавить или обновить строку в `knowledge/tasks/registry.md`.
3. Реализовывать изменения малыми обратимыми шагами; не смешивать unrelated GUI, сетевые и installer-изменения в одной задаче без явной причины.
4. После каждого изменения прогонять минимальные локальные проверки: `bash -n *.sh`, `bash -n libexec/*.sh`, `bash -n lib/*.sh`, `python3 -m py_compile gui/gui_server.py`, а для сетевых правок — ручной smoke по запуску, остановке и диагностике.
5. После завершения задачи обновить статус и итог в `task.md`, синхронизировать шаги и проверки в `plan.md`, затем обновить строку в `knowledge/tasks/registry.md` и только потом готовить commit/PR.
6. Если задача остановлена, признана лишней или заменена другой задачей, зафиксировать причину в `task.md` и `plan.md`, выставить статус `отменена` и отразить это в `knowledge/tasks/registry.md`.

Legacy-артефакты, перенесённые в `knowledge/tasks/*/artifacts/`, не использовать как отдельный рабочий контур и не переписывать задним числом без отдельной задачи на корректировку истории.

Если задача упирается в ручную сетевую проверку, это должно быть явно отражено и в `plan.md`, и в итоге `task.md`.

## Правила task-артефактов

Каждая задача в `knowledge/tasks/<TASK-ID>-<slug>/` должна содержать как минимум:

- `task.md` с паспортом задачи, статусом, целью, границами, контекстом, стратегией проверки и итогом;
- `plan.md` с конкретными изменениями по подсистемам, рисками, проверками, шагами и критериями завершения;
- строку в `knowledge/tasks/registry.md`, синхронизированную со статусом и кратким описанием задачи.

Для завершённых задач в `task.md` обязателен короткий блок `Итог` с фактически выполненными изменениями и остаточными рисками.
Для отменённых задач в `task.md` и `plan.md` обязателен короткий блок с причиной остановки или замены решения.

## Правила проверки

Автотестов сейчас нет, поэтому обязательна ручная smoke-проверка. Минимум: успешный старт через `run-xray-tun-subvost.sh`, наличие `tun0`, корректная остановка через `stop-xray-tun-subvost.sh`, и при изменениях GUI — открытие `http://127.0.0.1:8421` или `subvost-xray-tun.desktop` и выполнение команд `Старт`/`Стоп`/`Снять диагностику`. Открытие GUI не должно вызывать `pkexec`; root-запрос должен появляться только на действиях runtime. GUI backend считается внутренним компонентом и запускается через `open-subvost-gui.sh` или `subvost-xray-tun.desktop`, а не прямым пользовательским вызовом `libexec/start-gui-backend-root.sh`. Если меняете конфиги, приложите пример ошибки или лог из `logs/` при регрессии.

Если задача меняет preflight, DNS-логику, маршрутизацию или диагностику, дополнительно проверяйте негативные сценарии: отсутствие `tun`, конфликт DNS-менеджера, неполный старт `xray` runtime, корректность восстановления `resolv.conf`.

## Коммиты и Pull Request

История уже использует короткие префиксы: `feat:`, `docs:`, `chore:` и merge-коммиты вида `merge: topic/...`; придерживайтесь этого формата. Один PR должен решать одну задачу. В описании PR указывайте цель, затронутые сценарии запуска, ручные проверки и любые изменения в системных требованиях. Для изменений GUI прикладывайте скриншот, для сетевых правок — кратко опишите ожидаемое поведение DNS/TUN.

Если по задаче открыт task-контур, в PR нужно сослаться как минимум на каталог задачи и файл `knowledge/tasks/<TASK-ID>-<slug>/task.md`.

## Безопасность и конфигурация

В Git не должны попадать секреты, локальные адреса и диагностические дампы с приватными данными. `logs/*.log` и `__pycache__/` уже исключены из Git. `xray-tun-subvost.json` разрешён в репозитории только как санитизированный шаблон: поля `outbounds[0].settings.vnext[0].users[0].id`, `outbounds[0].streamSettings.realitySettings.publicKey` и `outbounds[0].streamSettings.realitySettings.shortId` должны оставаться placeholders `REPLACE_WITH_REALITY_UUID`, `REPLACE_WITH_REALITY_PUBLIC_KEY` и `REPLACE_WITH_REALITY_SHORT_ID`. Bundle стартует только из локально сгенерированного конфига выбранного узла. Любые изменения, требующие `sudo`, должны быть обратимыми и явно восстанавливать исходное состояние системы.

Скрипты не должны автоматически менять постоянную системную конфигурацию DNS, firewall и network-manager-сервисов без отдельной и явно документированной причины. По умолчанию допустимы только обратимые runtime-изменения и диагностические проверки.

## TASK FLOW

### Что считается задачей

- Любое изменение кода, конфигурации, схемы данных, интерфейсов, инфраструктурных настроек, тестов или поведения системы считается задачей.
- Любое изменение документации, если оно описывает изменение поведения системы, тоже считается задачей.
- Любое содержательное изменение продукта должно быть привязано к задаче.

### Что не считается задачей

- Сборка из исходников.
- Создание и сопровождение служебных скриптов задач.
- Технические операционные действия среды без изменения продукта.

Такие действия относятся к `knowledge/operations/`.

### Что не использовать

- Не создавать новые рабочие планы, черновики и task-артефакты в `.sisyphus/`.
- Для новой работы использовать только `knowledge/tasks/` и `knowledge/operations/`.

### Явная активация режима задачи

- Если в формулировке пользователя есть слово `Задача`, режим задачи должен быть активирован явно.
- Даже без слова `Задача` режим задачи обязателен, если запрос меняет код, конфигурацию, схему данных, интерфейсы, тесты или поведение системы.

### Где хранить задачу

- Каноническое место для задачи: `knowledge/tasks/<TASK-ID>-<slug>/`
- Источник истины по задаче: `task.md`
- Обязательные файлы задачи: `task.md`, `plan.md`
- `sdd.md` обязателен для сложных задач и живёт рядом с `task.md` и `plan.md`
- Дополнительные файлы: `worklog.md`, `decisions.md`, `handoff.md`, `artifacts/`, `subtasks/`
- Навигационный реестр задач: `knowledge/tasks/registry.md`

### Как открывать задачу

- Если запрос явно продолжает текущую активную задачу, использовать её как основной контекст.
- Если активной задачи нет, но запрос является задачей, открыть новую внутреннюю задачу с ID формата `TASK-2026-0001`.
- Если формулировка неоднозначна и может относиться и к текущей задаче, и к новой, нужно явно уточнить контекст у пользователя.
- При открытии новой задачи сразу создавать каталог задачи, `task.md`, `plan.md` и запись в `knowledge/tasks/registry.md`.
- Если задача сложная, при открытии сразу создавать и `sdd.md`; если сложность неоднозначна, нужно явно уточнить это у пользователя.

### Как автоматически выбирать между текущей задачей, подзадачей и новой задачей

- Если запрос укладывается в цель и границы активной задачи и не образует отдельный подконтур работы со своим `plan.md`, продолжать текущую задачу без создания новой сущности.
- Если запрос остаётся внутри цели активной задачи, но образует отдельный этап, компонент, интеграцию, исследование или проверку со своим жизненным циклом, создавать подзадачу внутри активной задачи.
- Если запрос меняет основную цель, продуктовый результат или должен сопровождаться независимо от текущей задачи, открывать новую задачу.
- Если пользователь явно указывает другой `TASK-ID`, переключать контекст на указанную задачу, а не создавать новую молча.
- Если пользователь явно просит подзадачу и у запроса есть корректный родительский контекст, создавать подзадачу.

### Что делать при неоднозначности маршрутизации

- Если после локального не-мутирующего исследования нельзя однозначно выбрать между продолжением текущей задачи, созданием подзадачи и открытием новой задачи, агент обязан остановиться и явно спросить пользователя.
- Вопрос должен кратко объяснять, в чём неоднозначность, и давать варианты действий.
- Предпочтительный формат вариантов: `A` — продолжить текущую задачу, `B` — создать подзадачу, `C` — открыть новую задачу.
- Если хост поддерживает structured user input, использовать его; иначе задавать короткий обычный вопрос, но всё равно с вариантами.

### Git-контур задачи

- Для верхнеуровневой задачи по умолчанию использовать отдельную ветку `task/<task-id-lower>-<slug>`.
- При открытии новой задачи на чистом рабочем дереве агент обязан сам создать или переиспользовать task-ветку и сразу синхронизировать поле `Ветка` в `task.md` и строку в `knowledge/tasks/registry.md`.
- Для подзадачи по умолчанию использовать ветку родителя; отдельную ветку создавать только если из локального контекста ясно, что подзадаче нужна изоляция.
- Если рабочее дерево грязное несвязанными изменениями, текущая ветка уже привязана к другой задаче или границы изменения неочевидны, автоматически не переключать ветку и сначала уточнить стратегию.
- После каждого смыслового этапа агент должен сам делать commit diff-а в пределах задачи с префиксом `TASK-ID`, если из локального контекста ясно, что изменения относятся к одной задаче и не содержат чужой незавершенной работы.
- Поле `Ветка` в `task.md` и колонка `Ветка` в `knowledge/tasks/registry.md` всегда должны совпадать с фактической рабочей веткой задачи.

### Когда git-действия не требуют подтверждения

- `git status`, `git branch --show-current`, `git diff --check` и другие безопасные проверки только на чтение.
- `git checkout -b` / `git switch -c` для task-ветки на чистом рабочем дереве.
- `git add` файлов задачи и `git commit` с префиксом `TASK-ID`, если границы изменения ясны.
- Без явного запроса пользователя не выполнять `git push`, `git rebase`, `git reset`, `git clean`, удаление веток и любое переписывание истории.

### Как обновлять старую версию навыка или knowledge-системы

- Перед upgrade сначала выполнить локальную проверку совместимости и определить точный scope перехода.
- Уже созданные task-каталоги, `task.md`/`plan.md`/`sdd.md` и `knowledge/tasks/registry.md` считать project data; не перезаписывать их молча при обновлении.
- Сам переход версии разносить в локальном git отдельным task-scoped commit-ом, чтобы момент перехода был виден однозначно.
- После transition-commit последующие рабочие изменения делать отдельными commit-ами, а не смешивать с самим переходом.
- Предлагать `push` только если через `git remote` или `git remote -v` реально обнаружен связанный remote; если remote нет, работать только локально и не поднимать тему `push`.

### Правило плана

- Одна задача = один `plan.md`
- Одна подзадача = один `plan.md`
- Глобальные каталоги `plans/` не использовать
- Если задача слишком большая, её нужно декомпозировать на подзадачи, а не строить отдельную общую дорожную карту
- `plan.md` должен ссылаться на этапы `sdd.md`, если SDD обязателен, но не должен дублировать всю спецификацию

### Идентификаторы

- Если есть внешний бизнес-ID, использовать его как основной идентификатор задачи
- Для внутренних задач использовать формат `TASK-2026-0001`
- Если для подзадачи нет отдельного явного ID, использовать наследуемый ID от родителя
- Если пользователь не указал явный ID подзадачи, допустимо использовать наследуемый ID вида `TASK-2026-0001.1`

### Подзадачи

- Подзадача хранится внутри `subtasks/` родительской задачи
- Подзадача повторяет ту же структуру, что и родитель
- Формально допустимо не более 3 уровней вложенности подзадач
- Если требуется 4-й уровень, нужно явно предупредить пользователя и предложить перепроектировать декомпозицию

### Контекст по умолчанию

- По умолчанию основным контекстом считается текущая активная задача
- Если пользователь явно указывает другой ID, приоритет у него
- Если формулировка допускает неоднозначность контекста, нужно коротко уточнить, какую задачу пользователь имеет в виду

### Когда SDD обязателен

- Если меняется схема данных, metadata, интеграционный контракт или публичное поведение между подсистемами, задача считается сложной и требует `sdd.md`
- Если изменение затрагивает несколько компонентов и должно идти по этапам с проверками, задача требует `sdd.md`
- Если ошибка не может быть надёжно отловлена одной локальной проверкой, задача требует `sdd.md`
- Если пользователь явно просит спецификацию, этапность или audit-gates, задача требует `sdd.md`
- Если сложность неочевидна, нужно коротко уточнить у пользователя, обязателен ли для задачи SDD
