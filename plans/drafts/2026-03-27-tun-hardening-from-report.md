# Заимствования из `docs/research/deep-research-report.md` для hardening bundle

- Дата: 2026-03-27
- Статус: planned
- Источник: `docs/research/deep-research-report.md`

## Цель

Усилить bundle за счёт полезных практик из исследовательского отчёта, не превращая проект в клон HAPP и не усложняя штатный сценарий запуска.

Ожидаемый результат: старт и диагностика дают более предсказуемое поведение при проблемах с TUN, DNS и маршрутизацией, а правила проекта фиксируют единый способ документирования таких задач.

## Изменения

- Усилить preflight в `run-xray-tun-subvost.sh`: проверка `/dev/net/tun`, ранняя диагностика конфликтов DNS-окружения, более явные сообщения об ошибках до старта `xray` и `sing-box`.
- Добавить post-start sanity check: после запуска `sing-box` проверять наличие `tun0`, ожидаемую автоконфигурацию маршрутов и понятный аварийный выход с автоматическим указанием пути к диагностике.
- Расширить `capture-xray-tun-state.sh`: добавить сведения о `/dev/net/tun`, `lsmod | grep tun`, `readlink -f /etc/resolv.conf`, состоянии `systemd-resolved` и `NetworkManager`, а также kernel/systemd-журналы по TUN.
- Обновить README коротким troubleshooting-разделом с фиксированным порядком проверки: TUN-устройство, интерфейс, маршруты, DNS, firewall, MTU.
- Не добавлять автоматическую настройку `iptables`, `nftables`, `ip_forward`, `setcap` или постоянную перенастройку системных сетевых сервисов. Эти вещи остаются только частью диагностики и документации.
- Оставить рабочие JSON-конфиги operator-managed; не строить реализацию вокруг их версионируемых миграций, поскольку они исключены из Git.

## Проверки

- `bash -n *.sh`
- `python3 -m py_compile gui/gui_server.py`
- Ручной happy-path: запуск через `sudo ./run-xray-tun-subvost.sh`, наличие `tun0`, успешная остановка через `./stop-xray-tun-subvost.sh`
- Ручной negative-path: понятная ошибка при проблеме с `tun` или DNS-preflight
- Проверка, что `capture-xray-tun-state.sh` формирует единый лог с новыми секциями диагностики

## Допущения

- Bundle остаётся обычным клиентом, а не gateway/router, поэтому автоматический NAT и `FORWARD` не внедряются.
- Rootless-режим не внедряется; штатный запуск по-прежнему идёт через `sudo` и `pkexec`.
- MTU по умолчанию не меняется автоматически; допускается только документированный fallback для ручной диагностики.
- Конфликты с `systemd-resolved` и `NetworkManager` в этой итерации решаются детектом, предупреждением и диагностикой, а не изменением системной конфигурации.
